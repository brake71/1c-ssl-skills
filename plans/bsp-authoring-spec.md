## 0. Главная цель

Reference учит **правильно применять** БСП в прикладной разработке: какие
модули/методы вызывать, с какими параметрами, какие нюансы и антипаттерны.
Скил **самодостаточен**: в выходе НЕ ссылается на `bsp3111_md/` и не содержит
хардкод-номеров строк `src/cf/...:NNN`. Параметры и примеры — inline.
Ссылка на исходник — только «модуль + метод» (агент сам найдёт строку через
`bsp_api.py` или греп в своём `src/cf`).

`bsp3111_md/` и `src/cf/` — **источники при авторинге**, в скил не входят.

## 1. Источники при авторинге (в порядке проверки)

1. **Старый reference** — `.claude/skills/bsp-<cluster>/references/bsp-<старое-имя>.md`
   (и компаньон `*-key-methods.md`, если есть): задачи, Anti-patterns, нюансы,
   список методов. Оттуда — каркас сценария и список методов.
2. **Дока БСП** — `bsp3111_md/*.md` (3046 файлов, плоско). Поиск:
   - подсистема/группа: `ls bsp3111_md/ | grep -i "<транслит_подсистемы>"` (напр.
     `базовая_функциональность`) → файл-описание → подряд файлы-методы;
   - конкретный метод: `ls bsp3111_md/ | grep -i "<транслит_метода>"` (напр.
     `сообщитьпользователю`). В дока-файле — секции `Синтаксис`, `Параметры`,
     `Пример вызова`, `Доступность`.
3. **Исходник** — `src/cf/CommonModules/<Модуль>/Ext/Module.bsl` —
   **авторитетен для сигнатур/регионов**. Через скрипт:
   `python .claude/skills/bsp/scripts/bsp_api.py method <Имя> [--module <Модуль>] --src src/cf`
   и `... module <Модуль> --src src/cf`.
4. **Если дока и исходник расходятся** — приоритет у исходника для сигнатур;
   для поведения/назначения сверяй оба. Расхождение — отмечай в «Нюансах».

## 2. Скрипт `bsp_api.py` — как пользоваться

- `method <Имя> [--module <Модуль>] --src src/cf` — модуль + полная сигнатура +
  **имя региона** + doc-комментарий (обрезанный) + путь.
  - Если метод в **нескольких модулях с разными сигнатурами** → скрипт выдаёт
    список и просит `--module <ИмяМодуля>`. ВСЕГДА уточняй нужный модуль через
    `--module` (канонический для подсистемы). Если метод в нескольких модулях с
    **одинаковой** сигнатурой (server/client вариант) — скрипт выведет детали
    базового + пометку «Также определён в …». Это нормально; в «Нюансах»
    укажи оба контекста (сервер/клиент).
- `module <Модуль> --src src/cf` — все экспортные методы модуля с сигнатурами +
  регион. Если модуль `*Переопределяемый` — скрипт помечает, что методы это хуки.
- `modules --src src/cf` — список модулей со stable API (для поиска кандидатов).

**Регион отдаётся именем** (`ПрограммныйИнтерфейс`/`СлужебныйПрограммныйИнтерфейс`/
`СлужебныеПроцедурыИФункции`/`УстаревшиеПроцедурыИФункции`/`ПереопределениеВызовов`).
Ты классифицируешь стабильность по имени региона.

## 3. Маркировка регионов/хуков (критично)

- `ПрограммныйИнтерфейс` → **stable**, метка не нужна (по умолчанию). Это то,
  что вызывает прикладной код.
- `СлужебныйПрограммныйИнтерфейс` / `СлужебныеПроцедурыИФункции` → ⚠️ служебный,
  обратная совместимость не гарантируется. В «Нюансах» — указать стабильную
  альтернативу; не давать как основной способ.
- `УстаревшиеПроцедурыИФункции` → ⚠️ **deprecated**: «устаревший, не
  использовать в новом коде; альтернатива — `<Метод>`».
- Модуль `*Переопределяемый` (хуки переопределения в регионе
  `ПрограммныйИнтерфейс`) → **хук**: «БСП вызывает этот метод, прикладной код
  РЕАЛИЗУЕТ его (копирует модуль-переопределитель в конфигурацию и
  переопределяет тело). НЕ вызывается из прикладного кода напрямую». НЕ давай
  как «вызови `Модуль.Метод(...)`».
- Реальные override-регионы `ПереопределениеВызовов` /
  `ПереопределениеТекстаЗапросаНабораДанных` — та же семантика хука (редки).

```

## Модули

(краткий список модулей группы с суффиксной логикой + ⚠️ несуществующие
модули/варианты, типовые ошибки выбора модуля.)

## Сценарии

### 1. <Глагол задачи>
... (4-10 сценариев по шаблону ниже)

(опц.) ## Редкие методы

(влитые из *-key-methods.md компаньона: краткое описание + сигнатура +
ключевой нюанс, без полного сценария. Или компактный список.)
```

## 5. Шаблон одного сценария (~25-40 строк)

```markdown
### <Номер>. <Глагол задачи>

**Задача:** <одной строкой>

**Функция:**
`<Модуль>.<Метод>(<полная сигнатура>) Экспорт`
— <Процедура/Функция>, регион: `#Область ПрограммныйИнтерфейс` (stable). <Сервер/Клиент/…>.

**Параметры:**
- `<Имя>` (<Тип>) — <назначение, дефолт, нюансы>
- ...

**Пример:**
​```bsl
<рабочий код вызова 3-8 строк с комментариями>
​```

**Нюансы / антипаттерны:**
- ❌ <типичная ошибка> — <почему / чем заменить>
- <нюанс поведения: фоновое задание, транзакция, RLS, привилегированный режим, …>
```

Правила:
- В **Функция:** — backtick вокруг `Модуль.Метод(полная сигнатура) Экспорт`.
  Полная сигнатура = из `src/cf` (через `bsp_api.py method --module`), со всеми
  параметрами и дефолтами. Не сокращай параметры.
- 4-10 сценариев на файл. Группируй родственные методы в один сценарий
  (напр. сериализация XML/JSON — один сценарий с 4 функциями).
- **Пример** — реальный рабочий код вызова, с комментариями. Не выдумывай
  имена параметров — бери из `src/cf`/`bsp3111_md`.
- **Нюансы** — конкретные типичные ошибки этой подсистемы (антипаттерны БСП) +
  поведенческие нюансы (фоновое задание, транзакция, RLS, привилегированный
  режим, модель сервиса, асинхронность).
