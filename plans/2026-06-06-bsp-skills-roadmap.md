# План подготовки скилов для AI-агента по работе с конфигурациями 1С:БСП 3.1.11

> **Назначение документа:** приоритезированный роадмап создания набора скилов (skills) для AI-агента, который пишет / рефакторит / ревьюит код на BSL в конфигурациях 1С, содержащих библиотеку стандартных подсистем 3.1.11.
>
> **Аудитория:** разработчик навыков (skill author) и LLM-агент, исполняющий план.

## 0. Quick-start для исполнителя (TL;DR)

> **Перед чтением 1000-строчного плана прочти это.** Это минимальный набор инструкций, без которого исполнитель натыкается на типовые грабли.

1. **Прочитай §3 (шаблон скилла) и §9 (DoD) ПЕРЕД началом работы.** Это обязательные разделы. §1–§2, §5–§7 — справочные, читай по необходимости.

2. **Перед Фазой 0 сделай §8.1 (Pre-flight)** — массовую сверку сигнатур через `src/cf/`. Без неё в плане останутся фантомные методы, которые исполнитель перенесёт в скилы как есть.

3. **Используй `.claude/skills/opencode-runner/references/test-harness.sh` для RED/GREEN-прогонов** (см. §9.4 и §9.5). Это автоматизирует 80% ручной работы: токены, итерации, skill_activated, method_accuracy, anti-pattern compliance. **Ручной Pattern 1** из opencode-runner — только для разовых отладок.
   - ⚠️ **КРИТИЧНО:** RED и GREEN **обязаны** прогоняться в **одной и той же `--dir`**. `test-harness.sh` это делает автоматически. **НЕ** запускайте RED вручную через `opencode run --dir <другая папка>` — окружения будут разные, сравнение невалидно (агент в RED увидит, например, `bsp3111_md/` и нагенерит leaks путей, в GREEN — не увидит, и «улучшение» будет артефактом смены окружения, а не эффектом скила).
   - **Дефолтное значение `--dir`:** `<repo>/src/` — там есть `src/cf/` (исходники конфигурации), но НЕТ `bsp3111_md/` (документация). Это симулирует реальный целевой репо агента.
   - ⚠️ **КРИТИЧНО (Bug 1 из `session-ses_161a.md`):** для устойчивых метрик на flaky-моделях (`9router/my-free` и др.) используйте `--runs 3` (majority vote по N прогонам). Single-shot `--runs 1` может выдать `activation=false` на нормальном скиле просто по неудаче дрейфа модели. См. `opencode-runner` §Key commands → `--runs N`.

4. **L0 — навигатор.** При запуске RED baseline для L0 **все 3 задачи пройдут** (case 2 из §9.0). Это **норма**, а не сбой. Skill оправдан, если приносит точную терминологию, suffix-систему, иерархию подсистем и навигационные patterns.
   - **Запускайте `test-harness.sh` с `--skill-type navigator`** (или оставьте `auto` — детектится по числу backtick-обёрнутых `Module.Method` в теле скила; для L0 < 3 → navigator). Это:
     - пропускает `check_compiles` в DoD (ответы L0 — prose, не BSL, `КонецПроцедуры` отсутствует — это норма);
     - включает 3-й путь активации (≥ 2 backtick-обёрнутых **имён модулей** из навигационной карты в ответе агента) — критично, потому что у L0 нет `Method.Name`, и 2-й путь активации не сработает.

5. **Path safety на Windows + Git Bash** (см. §9.5):
   - bash: `> /tmp/foo` создаёт файл в `C:\Users\<u>\AppData\Local\Temp\foo`
   - node: `fs.readFileSync('/tmp/foo')` ищет в `E:\tmp\foo` (current drive) — **не там**
   - **Решение:** используй workspace-local tmp (`./.bsp_test_tmp_$$/`) или `mktemp -d`. test-harness.sh уже это учитывает.

6. **Не правь** `bsp3111_md/`, `bsp321_md/`, `src/` (последний в `.gitignore`). Эти каталоги — только для чтения, **источник истины** при написании скилов (см. §1.3, §9.1 п.7).

7. **Один скил за итерацию.** Не пытайся сделать все 24 скила за заход — качество важно скорости. После каждой итерации — RED→GREEN→REFACTOR (см. §9), затем отметка чек-бокса (см. §14).
8. **При ошибках сначала читай §9.5 (Path safety), §11 (Открытые вопросы), §11.1 (Триггеры ревизии).** Большинство типовых проблем уже разобраны.

### 0.1. ⚠️ АНТИПАТТЕРНЫ ПРЕДЫДУЩЕГО ПРОГОНА (читать ОБЯЗАТЕЛЬНО перед стартом!)

> Разбор `session-ses_1616.md` (июнь 2026) — прогон агента, который **отметил чек-бокс [x] при некорректной валидации**. Чтобы следующий агент не повторил:

1. ❌ **Запускать 1 задачу × `--runs 1` и отмечать чек-бокс.** Это **прямое нарушение §9.0** («3 задачи из черновика `## When to use`»). Предыдущий агент сделал **один** запуск с **одной** задачей из-за таймаута 10 минут, и всё равно поставил `[x]`. **Запрет:** если не хватило времени на полный прогон — лучше **откатить скил** и оставить `[ ]`, чем отметить `[x]` без валидации. Таймаут сессии **не** оправдание для сокращения числа задач. **Рекомендация для таймаута:** `--runs 1` **допустим** для L0-navigator (Bug 8: 1 прогон ненадёжен, но §9.0 case 2 явно говорит, что L0 — refiner, не fixer, и 1 прогон = 3 в данном случае). **3 разных задачи** — обязательны, иначе проверять нечего.
2. ❌ **`description` в frontmatter содержит workflow-summary.** Агент вписал «Triggers: identifying which module owns a function, locating a subsystem's common modules, reading module region markers...» — это **антипаттерн CSO** (`writing-skills` §Claude Search Optimization): «NEVER summarize the skill's process or workflow». Корректно: только триггеры («Use this skill when the agent is asked X»). Workflow (как агент должен отвечать) — в `## When to use` и `## Core concepts`, **не** в `description`.
3. ❌ **Смешанный English + кириллица в `description`.** План §3.1.1: «`description` — на английском». Слово «ПрограммныйИнтерфейс» латиницей — нормально (транслит), но **русские слова кириллицей** в description — нет.
4. ❌ **Писать DoD-секции в `SKILL.md`.** Verification log, Baseline log, Smoke-test log, Self-check — это **служебная информация автора**, не потребитель скила. Скил — это **deliverable**, который копируется в целевой репо агента. Туда не должно попадать ничего про «как мы валидировали этот скил». Все DoD-доказательства пишутся в `reports/bsp-skills/<имя-скила>.md` (см. §9.6). Также: скил **не должен ссылаться** на этот файл.
5. ❌ **Игнорировать гайд-команды самого плана.** Все примеры с `//` как region marker заменены на `#Область` (исправлено в июне 2026 по итогам этого же прогона). Если видишь `//` как region marker в плане/скиле/документации — это **ошибка**, не цитируй её.
6. ❌ **Сокращать список ключевых модулей в `## Core concepts` из лени.** В §4 L0 карта «подсистема → модули» содержит **все 9-10 модулей** семейства (не только `ОбщегоНазначения*`). Сокращение карты до 2-3 модулей делает навигационный скил бесполезным — `bsp-base-common` (L1) начнёт заново собирать ту же карту.

> **Перед отметкой `[x]`** убедись, что в `reports/bsp-skills/<имя-скила>.md` заполнены: `## Self-check §9.1` (все 13 пунктов), `## Baseline log` (RED-фаза), `## Smoke-test log` (GREEN-фаза), `## Verification log` (сверки). Если хотя бы один раздел отсутствует или пуст — **не отмечай** `[x]`.

## 1. Контекст и ключевые принципы

### 1.1. Целевой сценарий

Агент работает в репозитории конкретной конфигурации 1С (содержащей исходники БСП как подсистему). Его задачи:

- написание нового кода на BSL с опорой на программный интерфейс БСП;
- рефакторинг/миграция существующего кода на стандартные вызовы БСП;
- консультирование и code review на соответствие стандартам БСП.

### 1.2. Абсолютные ограничения (self-contained)

- **В целевом репозитории агента нет файлов документации БСП** (`bsp3111_md/` не существует в его рабочем окружении) и **нет выгрузки исходного кода БСП** (`src/cf/` тоже отсутствует). Любые ссылки на `bsp3111_md/...` и `src/cf/...` внутри скила **бесполезны и запрещены**.
- Каждый скил — **самодостаточная инструкция**: в нём должны быть базовые принципы, ключевые методы с сигнатурами/примерами, паттерны и анти-паттерны.
- Скил может **ссылаться на исходный код БСП в репозитории конфигурации** (например, «посмотри реализацию `ОбщегоНазначения.ТекущаяДатаСеанса` в общем модуле `ОбщегоНазначения`»), но не на внешнюю документацию. Целевой репо агента — это конкретная конфигурация 1С, в которую БСП **уже включена** как подмножество, поэтому соответствующий BSL-код там физически присутствует.
- Для более глубокого изучения программного интерфейса агент **сам читает исходники** соответствующих общих модулей в репозитории конфигурации. Скил — это навигатор, а не копия документа.

### 1.3. Источник истины при разработке скилов (важное разграничение)

> **Сырьё при разработке ≠ содержимое готового скила.** AGENTS.md требует опираться на `bsp3111_md/` (это делается на этапе авторства), а §1.2 требует, чтобы **сам скил не содержал ссылок** на эти файлы. Документ `AGENTS.md` и принцип самодостаточности не противоречат: первый — про процесс, второй — про артефакт.

Документация в `bsp3111_md/` (3046 файлов, ~70 подсистем) и выгрузка исходного кода в `src/cf/` (271M, ~1600 общих модулей, только чтение) используются **только на этапе создания/аудита скилов**, как сырьё. После публикации скил от них не зависит и может применяться в репозитории, где ни `bsp3111_md/`, ни `src/cf/` нет вовсе.

**Приоритет источников при разработке скила** (если есть расхождение — сообщить пользователю, не выбирать молча):

1. **`src/cf/`** — исходный код БСП в XML/BSL. Авторитетный источник по факту существования метода, точной сигнатуре, типам параметров и возвращаемого значения. Проверять через `grep`/`Read` по `src/cf/CommonModules/<Имя>/Ext/Module.bsl`.
2. **`bsp3111_md/NNN_*.md`** — официальная документация. Использовать для понимания сценариев использования, примеров вызова, разделов «Пример вызова» и «Доступность». Может быть неполной или обтекаемой.
3. **Общие знания о платформе 1С** — только как последний источник, и **только** для тривиальных вещей (синтаксис языка, базовые типы).

> Согласно `AGENTS.md`: «При поиске факта: сначала документация БСП (`bsp3111_md/`), затем подтверждение по исходникам в `src/cf/`. Если нашли противоречие — сообщите пользователю». Мы **расширяем** этот порядок: при сверке сигнатур и факта существования метода `src/cf/` **приоритетнее**, поскольку является первоисточником. `bsp3111_md/` приоритетнее для сценариев использования и контекста.

Файлы-источники, на которые опираемся при создании скилов:

| Файл/каталог | Назначение при разработке |
|---|---|
| `bsp3111_md/001_…md` | Глава 1 — состав библиотеки, таблица подсистем с ключевыми функциями |
| `bsp3111_md/002_…md` | Глава 2 — инструкция по внедрению (НЕ приоритет для данных скилов) |
| `bsp3111_md/003_…md` | Глава 3 — настройка подсистем (НЕ приоритет) |
| `bsp3111_md/004_…md` | Глава 4 — список подсистем с API (оглавление) |
| `bsp3111_md/NNN_подсистема.md` | Заголовок подсистемы |
| `bsp3111_md/NNN_интерфейс.md` | Список API подсистемы (только имена, без описаний) |
| `bsp3111_md/NNN_имяфункции.md` | Описание одной функции (Синтаксис/Параметры/Возвращаемое значение/Пример/Доступность) |
| `src/cf/CommonModules/<Имя>/Ext/Module.bsl` | **Исходный код общего модуля** — для сверки сигнатур и существования метода |
| `src/cf/CommonModules/<Имя>/Ext/Module.xml` | Метаданные модуля (сервер/клиент/внешнее соединение, режим повторного использования) |
| `src/cf/Subsystems/<Имя>.xml` | Состав объектов подсистемы |

### 1.4. Соглашения по именованию и структуре скилов

- Имя скила: `bsp-<область>` (например, `bsp-base-common`, `bsp-users-access`).
- Расположение: `.claude/skills/bsp-<область>/SKILL.md` (рядом с уже существующими `agents-best-practices` и `prompt-crafting-guide`).
- Frontmatter — по образцу `agents-best-practices/SKILL.md`: `name`, `description`, `metadata`.
- `description` пишется **по триггерам задачи** («Use this skill when…»), а не по названию подсистемы.
- Минимум `## When to Use`, `## Core concepts`, `## Key methods`, `## Patterns`, `## Anti-patterns`, `## How to explore deeper` (см. §3).

## 2. Архитектура экосистемы скилов (4 слоя + Tooling)

| Слой | Назначение | Приоритет | Кол-во | Стратегия внедрения |
|---|---|---|---|---|
| **L0 — Foundation** | Базовый скил: «как вообще работать с БСП» — терминология, навигация по исходникам, конвенции | **P0** | 1 | Фаза 0, блокирует остальные |
| **L1 — Core** | Самые массовые подсистемы (вызовы встречаются в любой задаче) | **P0** | 6 | Фаза 1, после L0 |

**`opencode-runner`** — новый мета-скил, добавленный по итогам перепроверки плана. Содержит синтаксис `opencode run`, шаблон RED/GREEN-команд, метрики (A: токены/время, B: корректность вызовов, C: anti-pattern compliance, D: задача) с целевыми порогами, и готовый test-harness. **Использование обязательно** при выполнении §9.2 smoke-test (см. §9.4).

## 3. Формат (шаблон) одного скила

Каждый скил — **один файл `SKILL.md`** объёмом **≤ 500 строк / ≤ 5000 токенов** (рекомендация `prompt-crafting-guide`). Если превышает — таблицы и справочный материал выносятся в `references/`. Ниже — обязательная структура.

```markdown
---
name: bsp-<имя>
description: "Use this skill when the agent needs to <глагольные триггеры>, или <глагольные триггеры>. Do not use for <анти-триггеры>."
metadata:
  scope: bsp-3.1.11
  version: "3.1.11"
  layer: L1
  related_subsystems:
    - Подсистема1
    - Подсистема2
  related_skills:
    - bsp-fundamentals
    - bsp-base-common
---

# <Человекочитаемое название скила>

## When to use
- Триггер 1 (глагольная форма: «нужно отправить email», «требуется напечатать документ»)
- Триггер 2
- …

## Не использовать, если
- Анти-триггер 1 (например: «не использовать для доработки типовой конфигурации без БСП», «не использовать для задач вне БСП», «для базовых операций над строками/датами — см. `bsp-base-common`»)
- Анти-триггер 2
- …

## Core concepts
- Что такое подсистема, как устроена, какие общие модули её обслуживают
- Ключевые сущности и термины
- Где лежит исходный код в репозитории конфигурации (типичные имена общих модулей)

## Key methods
Таблица ключевых методов (10–20 строк для L1, 7–15 для L2, 5–10 для L3). Обязательные колонки:

| Метод | Сигнатура | Сервер/Клиент | Назначение | Пример вызова |
|---|---|---|---|---|

Сигнатуры в колонке «Сигнатура (ориентировочно)» приводятся **дословно** из `src/cf/CommonModules/<Имя>/Ext/Module.bsl` (приоритет) и/или `bsp3111_md/NNN_*.md` (дополнительно), **без ссылки** на файл в самом скиле (самодостаточность). Пометка «(ориентировочно)» — это **напоминание автору**, что перед публикацией скила сигнатура должна быть **дословно сверена** с источником; в финальном `SKILL.md` колонка остаётся с тем же названием, но **все ячейки заполнены конкретными сигнатурами** без `(уточнить)`.
**Запрещено** оставлять пометку `(уточнить)` в финальном скиле — это DoD-нарушение (§9 п.6).
Допустимо оставлять `…` только для **необязательных** параметров со значением по умолчанию, если в рамках скила они не нужны; в колонке «Пример вызова» — полная сигнатура.

## Patterns
3–5 типовых сценариев с минимальными примерами кода на BSL.
Каждый паттерн: «Когда нужно → код».

## Anti-patterns
5–8 явных «не делай так»:
- ❌ пример плохого кода
- ✅ как правильно

## How to explore deeper (с инструментами)
- **Какие общие модули/объекты метаданных** в репозитории конфигурации изучить для углубления (список конкретных имён).
- **Какие grep-шаблоны** применить (например, `grep -rn "^#Область ПрограммныйИнтерфейс" ОбщийМодуль.bsl`).
- **Какие подписки на события / определяемые типы** посмотреть в дереве метаданных.
- **На что обратить внимание в формах/командах** (типичные имена элементов `ПодменюПечать`, `ГруппаКомандОтчетов` и т. п.).
- Если архитектура сложна — допускается mermaid-диаграмма (это текст).
- **Навигация в целевом репо агента:** агент работает в выгрузке конфигурации 1С, куда БСП **уже включена** как подмножество — BSL-код модулей БСП физически присутствует в репо. Скил должен явно указывать: «найди общий модуль `<Имя>` (BSL/XML), открой область `#Область ПрограммныйИнтерфейс`, метод `<X>`».
```

### 3.2. Конвенция по «служебным» модулям

В коде БСП методы делятся на:

- **Программный интерфейс** (`#Область ПрограммныйИнтерфейс`) — стабильный, можно вызывать из прикладного кода.
- **Переопределение** (`#Область Переопределение`) — экспортные, но предназначены для замещения, а не вызова.
- **СлужебныйПрограммныйИнтерфейс** (`#Область СлужебныйПрограммныйИнтерфейс`) — экспортные, **обратная совместимость не гарантируется**.
- **СлужебныеПроцедурыИФункции** (`#Область СлужебныеПроцедурыИФункции`) — то же самое, ещё глубже внутреннее.

> **Правило для скилов:** методы из служебных модулей (`*Служебный.Метод()`) могут упоминаться в L1/L2 скилах **только с явной отметкой** ⚠️ «служебный API, может измениться» в колонке «Пример вызова» или сноской. В L0 (`bsp-fundamentals`) формулируется общее предупреждение. Если метод можно заменить стабильным аналогом — выбираем аналог.

## 4. L0 — Foundation

**Цель:** научить агента «видеть» БСП в репозитории конфигурации и не путаться в терминах.

### 5.6. `bsp-data-exchange` (**P0 → пересмотр: L2**)

> ⚠️ **Пересмотр L-уровня по итогам сверки:** из 8 методов в оригинальной таблице — **0 стабильных для прямого вызова** из прикладного кода. Переопределяемые/событийные методы — для **переопределения** поведения БСП, не для прямого вызова. Прямой API для регистрации изменений — `ПланыОбмена[ИмяПлана].ЗарегистрироватьИзменения` (платформенный, не БСП). По §10.1 критерию «> 80% стабильных методов» этот скил — **L2**, не L1. **Понижаем приоритет до P1**, переносим в Фазу 2.

**Подсистемы-источники:** Обмен данными, Обмен данными в модели сервиса, Синхронизация данных через универсальный формат (EnterpriseData).

**Why critical:** обмены — критическая инфраструктура, но **прямого API БСП для прикладного кода почти нет**. Агент скорее работает с **платформенными** механизмами (`ПланыОбмена`, `ОбменДанными.ОбменДаннымиXDTOСервер`) и с **настройкой правил** обмена (не с вызовом методов).

**Key methods (обязательный минимум):**

| Метод | Сигнатура (ориентировочно) | Сервер/Клиент | Назначение | Стабильность |
|---|---|---|---|---|

| `ОбменДаннымиСлужебный.ЗарегистрироватьИзменения` | (верифицировать) | Сервер | Регистрация изменений | ⚠️ служебный |
| `ОбменДаннымиСлужебный.УдалитьРегистрациюИзменений` | (верифицировать) | Сервер | Снять регистрацию | ⚠️ служебный |
| `ОбменДаннымиСлужебный.ВыполнитьОбменДанными` | (верифицировать) | Сервер | Запуск обмена | ⚠️ служебный |

**Patterns:**
- Добавление нового документа в существующий план обмена (правила / КД3 / EnterpriseData).
- Регистрация изменений по произвольному отбору.
- Обработка коллизий в обработчике `ПриКоллизииИзменений`.

**Anti-patterns:**
- ❌ Регистрировать изменения вручную через `РегистрыСведений` (есть `ПланыОбмена[ИмяПлана].ЗарегистрироватьИзменения` или `ОбменДаннымиСлужебный`).
- ❌ Делать обмен в транзакции пользовательской формы.
- ❌ Использовать прямое чтение XML вместо XDTO/EnterpriseData.

| `bsp-report-dedup` | Поиск и удаление дублей, Групповое изменение объектов | Массовая обработка данных |

**Формат для L3 — тот же шаблон §3, но допускается меньший объём (5–7 ключевых методов).**

### 7.1. Подсистемы с программным интерфейсом, не покрытые планом явно

Из **~70 подсистем верхнего уровня** (точное число варьируется в выгрузке — `.xml`-файлы в `src/cf/Subsystems/СтандартныеПодсистемы/Subsystems/`, включая вложенные 2-го уровня; проверить: `ls src/cf/Subsystems/СтандартныеПодсистемы/Subsystems/*.xml | wc -l`) следующие **не имеют своего скила** в плане — они либо отсутствуют в L1–L3, либо объединены с другими:

| Подсистема | Статус в плане | Рекомендация |
|---|---|---|
| `Регистрация` | ❌ не упомянута | Добавить в L3 при появлении задачи на программную регистрацию объектов |
| `ТранспортСообщенийОбмена` | ❌ объединена в `bsp-data-exchange` | OK — низкоуровневый транспорт редко вызывается из прикладного |
| `ПереводТекста` | ❌ не упомянута | Добавить в L3 (для задач мультиязычности / интеграции с переводчиками) |
| `КонтрольВеденияУчета` | ❌ не упомянута | Добавить в L3 (для задач валидации данных с произвольными правилами) |
| `СклонениеПредставленийОбъектов` | ❌ не упомянута | Добавить в L3 (для печатных форм и склонения ФИО) |
| `СтруктураПодчиненности` | ❌ не упомянута | Добавить в L3 (UI-фича, программный API минимален) |
| `ПолучениеФайловИзИнтернета` | ❌ не упомянута | Перенести в L2/L3 (часто нужна в обменах, но редко) |
| `ГенерацияШтрихкода` | ❌ не упомянута | Добавить в L3 (для задач печати этикеток) |
| `Взаимодействия` | ❌ не упомянута (пересекается с `bsp-comms`) | Объединить с `bsp-comms` ИЛИ выделить отдельно, если задачи на CRM-подсистему станут регулярными |
| `КонструкторФормул` | ❌ не упомянута | Добавить в L3 (UI-фича) |
| `РегламентныеЗадания` | ✅ в `bsp-longs-and-jobs` | OK — оставить в составе L1 |
| `БазоваяФункциональность` | ✅ частично в `bsp-base-common` + `bsp-longs-and-jobs` | OK — функциональные блоки разделены |

> **Правило:** новые скилы добавляются **только при появлении реальной задачи** (Фаза 3). Список выше — не «обязательство», а **backlog** для возможного расширения.

## 8. Фазы выпуска и приоритеты

> ⚠️ **Исправление к предыдущей версии:** фазы идут **строго последовательно**. L1 явно опирается на терминологию и конвенции L0, поэтому запускать L1 «параллельно» с L0 нельзя.

| Фаза | Содержимое | Зависимости | Критерии готовности фазы |
|---|---|---|---|
| **Фаза -1 (Pre-flight)** | Массовая сверка сигнатур всех таблиц Key methods плана против `src/cf/` и `bsp3111_md/`. Генерация «verified»-таблицы для каждого скила. См. §8.1. | — | Каждая сигнатура в §5–§6 либо помечена как ✅ (сверена и совпадает), либо как ⚠️ (нужно исправить). Нет ❌ (выдуманных/несуществующих). |
| **Фаза 0** | L0: `bsp-fundamentals` | — | Скил проходит self-check: описание содержит триггеры, нет ссылок на внешние доку, перечислены основные термины и конвенции (в т.ч. про служебные модули) |
| **Фаза 1** | L1: 5 скилов (5.1–5.5) — `bsp-data-exchange` перенесён в L2 по результатам сверки (см. §5.6) | **требует Фазу 0** | Каждый скил покрывает 10–20 ключевых методов, имеет 3+ Patterns и 5+ Anti-patterns, инструкцию «How to explore deeper», smoke-test по §9 |
| **Фаза 2** | L2: 8 скилов (5.6 + 6.1–6.7) | **требует Фазу 1** | Аналогично Фазе 1 |
| **Фаза 3** | L3: 10 скилов — по запросу | **требует Фазу 2** | Делаем **только** когда появляется реальная задача из этой подсистемы |
| **Фаза 4** | Аудит всех опубликованных скилов на согласованность имён, ссылок, терминов. Явные критерии — см. §8.2. | **требует все предыдущие** | 7 проверок §8.2 пройдены |

**Внутри фазы 1 и 2** скилы между собой не зависят и могут создаваться в произвольном порядке (по одной итерации на скил).

**Рекомендация по итерациям:** не пытаться сделать все 24 скила за один заход — качество важно. Один скил за итерацию, с ревью (см. §13).

### 8.1. Фаза -1 (Pre-flight): массовая сверка сигнатур

> **Зачем:** без этой фазы исполнитель будет тратить основное время не на написание скила, а на сверку каждой сигнатуры (а их ~100). Выявленные в финальной ревью ошибки (`ТекущаяДатаСеанса` не экспорт, `БезопасноеХранилище.*` не существует, `СообщитьПользователю` — 5 параметров не 4) — типичные примеры.

**Задачи фазы:**

1. **Автовыгрузка сигнатур из `src/cf/`:** для каждого модуля из §5–§6 извлечь список всех экспортных функций и процедур.
   ```bash
   for module in ОбщегоНазначения ОбщегоНазначенияКлиент СтроковыеФункцииКлиентСервер \
                ПодключаемыеКоманды ДлительныеОперации РегламентныеЗаданияСервер \
                УправлениеПечатью УправлениеКонтактнойИнформацией; do
     grep -E "^(Функция|Процедура) [А-ЯA-Z][А-ЯA-Za-z_]+\(" \
       "src/cf/CommonModules/$module/Ext/Module.bsl" 2>/dev/null \
       | sed -E 's/^(Функция|Процедура) ([А-ЯA-Za-z_]+).*$/\2/' \
       | sort -u
   done > /tmp/bsp_exports.txt
   ```
2. **Построчная проверка таблиц §5–§6** — для каждого метода в Key methods выполнить:
   ```bash
   for entry in "Модуль.Метод" "Модуль.Метод" ...; do
     mod="${entry%%.*}"; mtd="${entry##*.}"
     grep -q "^Функция $mtd\|^Процедура $mtd" \
       "src/cf/CommonModules/$mod/Ext/Module.bsl" 2>/dev/null \
       && echo "✅ $entry" || echo "❌ $entry"
   done
   ```
3. **Результат** — таблица в `plans/preflight-verification.md` с пометками ✅/⚠️/❌ для каждого метода. Любой ❌ — баг в плане, исправляется до старта Фазы 0.
4. **Платформенные методы** (нет в `src/cf/`, но валидны: `ТекущаяДатаСеанса`, `ПланыОбмена.ЗарегистрироватьИзменения`) — помечаются ⚙️ платформенный, **не** ❌.
5. **Критерий готовности фазы:** все строки в `preflight-verification.md` — либо ✅, либо ⚙️.

**Трудоёмкость:** ~30 минут (одноразовая задача).

### 8.2. Фаза 4 (Аудит): явные критерии перекрёстной консистентности

Аудит — это **не «проверить, что скилы существуют»** (это делает чек-лист), а **«согласованы ли скилы между собой»**. 7 проверок:

1. **Единые имена общих модулей** — для каждого модуля, упомянутого хотя бы в 2 скилах, имя везде одинаковое (без опечаток типа `ДлительныеОперацииСлужебный` vs `ДлительныеОперацииСервер`).
2. **Единые суффиксы контекста** — `Клиент` / `Сервер` / `КлиентСервер` / `ВызовСервера` — везде используются по одному правилу.
3. **Единый формат примеров кода BSL** — отступы (таб vs пробелы), выравнивание, стиль `Знач` / без `Знач` — согласованы.
4. **Согласованные триггеры в `## When to use`** — если один скил покрывает X, а другой Y, нет пересечения по триггерам (иначе агенту сложно выбрать).
5. **Перекрёстные ссылки** — каждый `related_skills` указывает на **существующий** скил.
6. **Отсутствие дублирования Key methods** — один и тот же метод не описан в двух скилах (кроме случаев типа `БезопасноеХранилище`, упомянутого в L0 и L1).
7. **Консистентность Anti-patterns** — если «не используй `ТекущаяДата()`» — это правило для всех скилов, не только для `bsp-base-common`.

**Критерий готовности фазы:** все 7 проверок пройдены, расхождения зафиксированы и исправлены.

## 9. Критерии приёмки (Definition of Done) для одного скила

Скил считается готовым, **когда выполнены ВСЕ пункты**. Применяется подход **RED → GREEN → REFACTOR** из `writing-skills`.

### 9.0. RED-фаза (baseline) — ОБЯЗАТЕЛЬНО до написания скила

- Берём **3 задачи** из черновика `## When to use` (первые 3 триггера).
- Прогоняем **одним LLM-агентом** **без скила** (только его общий контекст + описание БСП как пара абзацев).
- Фиксируем baseline: какие методы вызвал агент, были ли «свои» реализации вместо API БСП.
- Сохраняем в `## Baseline log` файла `reports/bsp-skills/<имя-скила>.md` (после GREEN-фазы).

> ⚠️ **КРИТИЧНО (Bug 4 из `session-ses_161a.md`):** RED и GREEN **обязаны** прогоняться в **идентичной `--dir`**. Используйте `test-harness.sh` — он сам делает RED+GREEN в одной `--dir` (см. §9.4). **НЕ** запускайте RED вручную через `opencode run --dir <другой путь>` — окружения будут разные:
> - в `--dir <repo-root>` агент видит `bsp3111_md/` и может «лик» путями в ответе;
> - в `--dir <repo>/src/` агент видит только исходники конфигурации, как и в боевом окружении.
>
> Если запустить RED в первом, а GREEN во втором — «улучшение» (нет лика путей в GREEN) будет **артефактом смены окружения**, а не эффектом скила. Сравнение бесполезно.
>
> **Дефолтное `--dir` для всех RED/GREEN-прогонов:** `<repo>/src/` (есть `src/cf/`, нет `bsp3111_md/` — это и есть симуляция целевого репо агента).
>
> Если **очень нужен** ручной RED для отладки одной задачи: используйте ту же `--dir`, что будет в GREEN, и удалите скил из `<dir>/.claude/skills/<skillname>/` перед запуском (иначе skill auto-discovery подцепит остатки прошлого GREEN).

> ⚠️ **КРИТИЧНО (Bug 8 из `session-ses_161a.md`):** модель `9router/my-free` (и любая другая) **не детерминирована**. Single-shot RED/GREEN (`--runs 1`) может выдать ложно-отрицательный `activation=false` просто по неудаче дрейфа. Для финального DoD-прогона используйте `--runs 3` (majority vote). См. `opencode-runner` §Key commands → `--runs N`.

**Кейс: baseline уже корректен (0 «своих» реализаций во всех 3 задачах).** Это означает одно из трёх:
1. Задачи слишком тривиальны и не «вытаскивают» из агента неправильные паттерны → **расширить задачи**: добавить угловые случаи (например, для `bsp-base-common` — задачу на форматирование строки с большим числом параметров, где агент мог бы скатиться в `СтрШаблон` или конкатенацию; для `bsp-print-reports` — задачу на нестандартный менеджер печати).
2. Базовые знания агента уже покрывают подсистему → скил **уточняет** паттерны (имена методов, тонкости сигнатур, ⚠️-служебные API), а не исправляет ошибки. **Это валидный исход** — скил оправдан, но baseline стоит обогатить нетривиальной задачей.
3. Скил действительно не нужен → признать и **не публиковать**. Сэкономленные усилия направить на более проблемную подсистему.

> **⚠️ Для L0 (навигатор) — case 2 это НОРМА, не edge case.** У L0 нет API-методов для проверки; агенту нечего сломать. Все 3 навигационные задачи **пройдут** на «голой» агенте. Это **не сбой плана** — это значит, что задачи слишком общие (любой LLM знает базовые подсистемы). Ожидаемый эффект L0-скилла:
> - **точная терминология** (region markers `#Область` НЕ `//`, suffix-система, `Переопределяемый` — для переопределения, а не вызова);
> - **навигационные patterns** («как за 3 шага найти нужный модуль», «как отличить стабильный API от ⚠️-служебного»);
> - **карта подсистем** (subsystem → primary modules, для навигации в L1-скилах);
> - **anti-patterns** (НЕ `БСП_` префикс, НЕ вызов `Переопределение`-модулей).
>
> Если baseline уже корректен — **расширь задачи**: добавь «ловушки», на которых LLM ошибается без скилла. Например, для `bsp-fundamentals`: «чем отличается `#Область` от `//`-комментария, и какой из них парсит препроцессор?»; «что означает суффикс `Служебный` — и в чём разница со стабильным API?». Зафиксируй baseline для 3+ таких задач, и если агент ошибается хотя бы в одной — скил оправдан по case 1. Если не ошибается — оправдан по case 2.
>
> **Для L0 в `test-harness.sh` обязательно указывайте `--skill-type navigator`** (или оставьте `auto` — детектится по числу backtick-обёрнутых `Module.Method` в теле, для L0 их < 3 → navigator). Это пропускает `check_compiles` в DoD (L0 даёт prose, не BSL) и включает 3-й путь активации (по именам модулей, не по `Module.Method`).

Решение принимается **до** начала GREEN-фазы, фиксируется в `## Baseline log` файла `reports/bsp-skills/<имя-скила>.md` отдельной строкой.

### 9.1. GREEN-фаза (написание скила) — все пункты должны быть ✅

1. ✅ Файл `.claude/skills/bsp-<имя>/SKILL.md` существует.
2. ✅ Frontmatter валиден: `name`, `description` (только триггеры, ≤ 1024 символа, начинается с `Use this skill when…`), `metadata.scope = bsp-3.1.11`, `metadata.version = "3.1.11"`, `metadata.layer` указан; `metadata.related_subsystems` и `metadata.related_skills` заполнены.

4. ✅ Раздел `## Не использовать, если` содержит **минимум 2** анти-триггера (что НЕ должен решать этот скил, на какой скил переключиться).
5. ✅ Раздел `## Core concepts` объясняет архитектуру подсистемы, имена общих модулей и карту «подсистема → модули» для затронутых подсистем.
6. ✅ Таблица `## Key methods` содержит **10–20 строк** (для L1), **7–15** (для L2), **5–10** (для L3), с обязательными колонками `Метод | Сигнатура | Сервер/Клиент | Назначение | Пример вызова`. Для служебных модулей — отдельная колонка `Стабильность` с пометкой ⚠️. **Пометка `(уточнить)` запрещена** — все сигнатуры должны быть сверены с `src/cf/` до публикации.
7. ✅ Каждая строка `Key methods` **дословно сверена** по двум источникам (приоритет — `src/cf/`, дополнительно — `bsp3111_md/NNN_*.md`):

   - запись сверки в виде `<!-- verified: src/cf/CommonModules/<…>/Module.bsl:N -->` — только в отчёте, **не** в `SKILL.md` (см. §3.1).
8. ✅ Раздел `## Patterns` содержит **минимум 3** примера с кодом на BSL.
9. ✅ Раздел `## Anti-patterns` содержит **минимум 5** пар «❌ / ✅».
10. ✅ Раздел `## How to explore deeper` (для L1/L2) указывает: имена общих модулей + минимум 1 grep-шаблон + минимум 1 glob-маску + (опционально) mermaid-диаграмму. Для L3 — допускается **только** имена модулей + 1 grep-pattern (без glob/mermaid).

### 9.2. Smoke-test (один агент, один прогон, как в `writing-skills`)

> ⚠️ **Упрощение по сравнению с предыдущей версией:** RED-GREEN-REFACTOR с одним агентом достаточно. Ранее предложенный критерий «2 агента × 3 задачи = 6 прогонов, ≥ 67% успеха» — избыточен для 24-скиловой экосистемы и на практике не будет выполнен.

> **Связь с §9.4 (количественные метрики):** п. 9.2 даёт **качественные** критерии («все 3 задачи компилируются», «вызовы из Key methods»). П. 9.4 — их **количественная** версия: `method_accuracy ≥ 0.5` (мин) / `≥ 0.9` (целевой), `skill_grounding ≥ 0.5` / `≥ 0.7`, `anti_pattern_violations = 0`, и т. д. На этапе GREEN-фазы §9.2 проверяется агентом-человеком (быстро), §9.4 — автоматизированно через `opencode run` (см. `opencode-runner` skill + `references/test-harness.sh`). В финальный DoD-зачёт скила идёт **§9.4** (объективно), §9.2 — sanity-check.

**Шаги ручного smoke-теста** (с проверкой `skill_activated`):

1. Поставить скил в `<рабочий-каталог>/.claude/skills/<skillname>/SKILL.md` (либо использовать `test-harness.sh --keep-staging`).
2. Открыть интерактивный TUI (`opencode`) **в этом каталоге**.
3. Дать агенту первую задачу из When to use.
4. **Проверить активацию:** в блоке `<think>...</think>` ответа агента должно быть явное упоминание скила по имени ИЛИ ссылки на него (например, «согласно скилу bsp-base-common, использую...»). Если агент сгенерировал код «из головы» **без ссылки на скил** — скил не активировался, либо его `description` не сработало. Это **провал §9.2**, даже если код правильный.
5. Повторить для задач 2 и 3.
6. Зафиксировать в `## Smoke-test log` файла `reports/bsp-skills/<имя-скила>.md`: список задач, имя агента, дата, **факт активации** (yes/no для каждой), краткий вердикт.

**Критерии прохождения (все должны быть ✅):**

- [ ] Все 3 задачи дают BSL-код, который (a) синтаксически компилируется (эвристика), (b) содержит вызовы **из Key methods таблицы скила** (а не «свои» реализации).
- [ ] Скил **активировался** хотя бы в 2 из 3 задач (агент ссылается на скил в `<think>` или в комментариях к BSL).
- [ ] Сравнение с baseline: количество вызовов «своих» реализаций **снизилось** до 0 хотя бы в 1 из 3 задач.

**Если хотя бы один пункт ❌ — REFACTOR-фаза:** усилить `description` (триггеры не сработали), добавить anti-patterns (агент сгенерировал «своё»), или уточнить Patterns (агент не нашёл нужный паттерн). Повторить прогон.

> ✅ **После прохождения smoke-test (зелёный результат):** агент **обязан** отметить чек-бокс в `plans/2026-06-06-bsp-skills-checklist.md` (см. §14) и обновить прогресс-сводку. Без этой отметки скил считается «не закрытым по DoD», даже если `SKILL.md` уже написан.

### 9.3. REFACTOR-фаза (закрытие лазеек)

- Прочитать результаты baseline + smoke-test.
- Найти **систематические** ошибки агента (одна и та же лазейка проявляется в 2+ задачах).
- Закрыть их: добавить в Anti-patterns, усилить Patterns, добавить пояснения в Core concepts.
- Не «тюнить» под конкретную задачу — только под класс ошибок.

### 9.4. Автоматизированный smoke-test через `opencode run` (рекомендуемый)

§9.0–9.3 описывают методологию. **Конкретный инструментарий** для её выполнения — мета-скилл `opencode-runner` (см. `.claude/skills/opencode-runner/SKILL.md`). Он покрывает:

- синтаксис `opencode run` (позиционные аргументы, флаги `--model`/`--format json`/`--dir`/`--pure`);
- **staging-подход** (скил ставится в `<dir>/.claude/skills/<skillname>/`, не через `--file`) — единственный способ проверить, что opencode реально активирует скил по `description`;
- список метрик (A: токены/время, B: корректность вызовов, C: anti-pattern compliance, D: задача) с целевыми порогами, **включая `skill_activated`**;
- парсинг JSON-вывода через `node` (без зависимости от `jq`);
- готовый скрипт `references/test-harness.sh`.

**Минимальный прогон (RED → GREEN) через staging:**

```bash
# Очистка staging
rm -rf "$WORKDIR/.claude/skills"   # WORKDIR = каталог с исходниками (например, src/)

# RED: пустой staging — скилов нет
opencode run "$TASK" \
  --model "$MODEL" \
  --format json \
  --dir "$WORKDIR" \
  --pure \
  --dangerously-skip-permissions \
  > /tmp/red.json

# GREEN: ставим скил в staging, opencode сам подцепит по auto-discovery
mkdir -p "$WORKDIR/.claude/skills/$SKILL_NAME"
cp "$SKILL_PATH" "$WORKDIR/.claude/skills/$SKILL_NAME/SKILL.md"
opencode run "$TASK" \
  --model "$MODEL" \
  --format json \
  --dir "$WORKDIR" \
  --pure \
  --dangerously-skip-permissions \
  > /tmp/green.json

# Cleanup
rm -rf "$WORKDIR/.claude/skills"
```

> ⚠️ **Не используйте `--file` для GREEN-прогона.** Это форсирует контекст, минуя auto-discovery. Скил с плохим `description` пройдёт GREEN-фазу тихо — агент просто не активирует его, но код всё равно сгенерирует «из головы». Без метрики `skill_activated` это невозможно отловить.

> **✅ По умолчанию — используйте `references/test-harness.sh`** (готовый скрипт opencode-runner). Это **не рекомендация, а дефолт**: ручной Pattern 1 выше нужен только для **разовых отладок** (проверить один метод, посмотреть как выглядит вывод). Для **DoD-прогона** — всегда test-harness. Он делает всё это автоматически и печатает 4 diff-таблицы + DoD-проверку одной командой (см. opencode-runner Pattern 5). Уже решает §9.5 (path safety).

> ⚠️ **Обязательные флаги для DoD-прогона** (по итогам разбора `session-ses_161a.md`):
> - `--runs 3` (или `--runs 5`) — majority vote против дрейфа модели (Bug 8). Single-shot `--runs 1` даёт ложно-отрицательные `activation=false` на нормальных скилах.
> - `--skill-type navigator` — для L0-скилов (или `auto` — детектится). Пропускает `compiles` в DoD и включает 3-й путь активации по именам модулей. Для L1/L2 — оставлять `auto` (детектится в `implementation`).
> - `--dir <repo>/src/` — НЕ корень репо. В корне есть `bsp3111_md/`, которая попадает в контекст агента и портит сравнение RED↔GREEN. Используйте `<repo>/src/` (там только исходники конфигурации, как в боевом окружении). Bug 4.
> - `--bsl-src <repo>/src/cf` — обязательно если `--dir` не корень репо с `src/cf/`. Без этого `method_accuracy` будет 100% missing.

**Целевые пороги** (зелёный DoD):

| Метрика | Минимальный (первая итерация) | Целевой (зрелый скил) | Применимость |
|---|---|---|---|
| `skill_activated` (bool, majority по `--runs N`) | `true` в GREEN, `false` в RED | то же | Все типы скилов |
| `method_accuracy` (correct / total) | ≥ 0.5 | ≥ 0.9 | Только `implementation`; для `navigator` `0/0` — OK |
| `hallucinated_methods` | 0 | 0 | Только `implementation` |
| `skill_grounding` (методы из Key methods / total) | ≥ 0.5 | ≥ 0.7 | Только `implementation` |
| `total_tokens` дельта vs baseline (median по N runs) | ≤ +30% | ≤ 0% (идеал) | Все |
| `compiles` (эвристика) | true | true | Только `implementation`; для `navigator` пропускается |
| `anti_pattern_violations` | 0 | 0 | Все |

> **Логика «мин vs целевой»:** первая итерация скила — высокие требования неразумны (скил ещё сырой, тестовые задачи могут не покрывать все аспекты). Стартуем с прагматичных порогов. По мере REFACTOR-фазы и публикации новых итераций — **повышаем** пороги в `references/test-harness.sh` (порог в скрипте) И в §9.4 (порог в плане). **SKILL.md §B хранит оба** как ориентир.

Если хотя бы один порог нарушен — переход в REFACTOR-фазу (см. §9.3) → повторный прогон.

**Фиксация результата** в `## Smoke-test log` файла `reports/bsp-skills/<имя-скила>.md`:

```markdown
## Smoke-test log
- Дата: 2026-06-15
- Opencode version: <из opencode --version>
- Model: 9router/my-free
- RED: baseline.json (input=N, output=N, total=N, iterations=N)
- GREEN: with_skill.json (input=N, output=N, total=N, iterations=N)
- method_accuracy: 0.95 (19/20)
- skill_grounding: 0.80 (16/20)
- anti_pattern_violations: 0
- total_tokens_delta: +12%
- compiles: true
- functional_complete: true
- Решение: ✅ DoD пройден → чек-бокс отмечен
```

> **Зачем автоматизация:** ручной прогон RED/GREEN «по памяти» через интерактивный TUI даёт **субъективную** оценку (агент «вроде бы вызвал правильный метод»). Только зафиксированные метрики + воспроизводимая команда дают основание отметить чек-бокс в `plans/2026-06-06-bsp-skills-checklist.md`. Это требование §14.1.

### 9.5. Path safety при RED/GREEN на Windows + Git Bash

> **Этот баг был реально пойман в первой сессии** (см. `session-ses_161d.md` строка 3722: `Error: ENOENT: no such file or directory, open 'E:\tmp\red_t1.json'`). Поэтому — отдельный раздел.

**Проблема:** на Windows + Git Bash пути `/tmp/*` интерпретируются **по-разному** разными процессами:

| Процесс | Как видит `/tmp/foo` | Где физически |
|---|---|---|
| bash (`> /tmp/foo`) | `C:\Users\<u>\AppData\Local\Temp\foo` | Windows temp |
| Node.js (`fs.readFileSync('/tmp/foo')`) | `E:\tmp\foo` (current drive) | `E:\tmp\` (обычно не существует) |
| opencode (Node-based) | То же, что Node.js | То же |

**Симптом:** `opencode run ... > /tmp/red.json` создаёт файл в Windows temp. Когда затем `node -e "fs.readFileSync('/tmp/red.json')"` пытается его прочитать — **файл не найден** (ENOTEMPTY/ENOENT).

**Решения (в порядке предпочтения):**

1. **`test-harness.sh` уже решает эту проблему** через workspace-local tmp: `TMP="$DIR/.bsp_test_tmp_$$"`. Это безопасно потому, что и bash, и opencode интерпретируют относительные пути одинаково.
2. **Workspace-local вручную:** `mkdir -p ./tmp_red_green && opencode run "..." > ./tmp_red_green/red.json`. Потом `node -e "fs.readFileSync('./tmp_red_green/red.json')"`.
3. **POSIX-портабельный tmp:** `mktemp -d -p .` создаёт уникальную папку в текущей директории (POSIX) — работает в bash на Windows.
4. **`%TEMP%` через `cmd //c`** — только если нужна совместимость с чисто-Windows инструментами: `TMP=$(cmd //c "echo %TEMP%" | sed 's|\\\\|/|g')`.

**Чего НЕ делать:**
- ❌ `opencode run > /tmp/foo` + `node -e "fs.readFileSync('/tmp/foo')"` — сломается на Windows.
- ❌ Передавать абсолютный Windows-путь (`E:\foo\bar.json`) в opencode на *nix-системах.

### 9.6. Хранение DoD-доказательств (вне `SKILL.md`)

> **Принцип:** `SKILL.md` — это **deliverable**, который копируется в целевой репо агента вместе с другими скилами. Всё, что нужно для **аудита качества скилов в этом репозитории** (Baseline log, Self-check, Smoke-test log, Verification log) — в отдельной папке, описанной ниже.

**Расположение:** `reports/bsp-skills/<имя-скила>.md` — по одному файлу на скил. Папка `reports/bsp-skills/` создаётся один раз в корне репозитория, **не** в `.claude/skills/`, **не** в `vault/` (vault/ в `.gitignore`).

## 10. Что НЕ делать (анти-паттерны процесса)

- ❌ Писать скил, копируя содержимое `bsp3111_md/` слово в слово.

- ❌ Включать в L1/L2 скилы про подсистемы, у которых **нет стабильного программного интерфейса** (только служебные модули) — такие подсистемы автоматически идут в L3.
- ❌ Создавать L3-скилы «на будущее» — только под реальную задачу.
- ❌ Игнорировать существующие `agents-best-practices` и `prompt-crafting-guide` — формат, frontmatter, чек-листы берём оттуда.
- ❌ Менять файлы в `bsp3111_md/` и `bsp321_md/` (это выгрузка документации, не наша зона ответственности, см. `AGENTS.md`).
- ❌ Тестировать скилы на синтетических задачах без привязки к 1С: проверять надо на реальных фрагментах BSL.
- ❌ Запускать фазы параллельно (L0 + L1, L1 + L2 и т. п.) — см. §8.
- ❌ Публиковать скил, в котором > 20% методов в Key methods помечены ⚠️ «служебный» — пересмотреть дизайн скила, скорее всего подсистема не предназначена для прямого вызова из прикладного кода.

### 10.1. Критерий отнесения подсистемы к слою

- **L0** — навигационные/терминологические знания, не привязанные к конкретной подсистеме.
- **L1** — подсистемы, для которых **в сумме по всем подсистемам, покрываемым одним скилом**:
  - в `bsp3111_md/NNN_интерфейс.md` (или в `src/cf/CommonModules/`, если файла интерфейса нет) — **≥ 30 функций** стабильного программного интерфейса;
  - есть типовые сценарии вызова в **каждой** прикладной конфигурации на БСП;
  - **> 80%** методов в Key methods — стабильные (не служебные).
  - **Пример:** `bsp-commands-external` покрывает две подсистемы (`ПодключаемыеКоманды` + `ДополнительныеОтчетыИОбработки`) — у каждой может быть < 30 функций индивидуально, но **в сумме ≥ 30**, и скил попадает в L1.
- **L2** — подсистемы, для которых (аналогично, в сумме):
  - **7–30 функций** стабильного программного интерфейса;
  - сценарии встречаются **регулярно**, но не в каждом проекте (60–80% стабильных методов).
- **L3** — всё остальное, **включая** подсистемы с преимущественно служебным API.

Конкретные скилы могут переезжать между слоями по мере накопления опыта (см. §11).

## 11. Открытые вопросы

- **Версионирование скилов при обновлении БСП.** В репозитории есть `bsp321_md/`, но 3.1.11 — основная. Нужен ли механизм «этот скил проверен для 3.1.11, для 3.2.1 — отдельно»? Пока фиксируем `metadata.version: "3.1.11"`.
- **Локализация описаний в frontmatter.** Делать `description` только на английском (чтобы триггеры срабатывали независимо от языка запроса), а `## When to use` — на русском, или полностью на русском? По умолчанию: `description` — на английском, тело — на русском, примеры кода — на русском BSL.
- **Связи между скилами.** `related_skills` в frontmatter — **разрешён** (см. §3.1, изменение по итогам ревью). Указывать только реально необходимые скилы (не более 3–4 на скил).
- **Применимость конкретных подсистем к L1/L2.** Реальные сценарии «встречается в каждой прикладной» — это гипотеза автора плана; первый реальный проект покажет, нужно ли перемещать скил в L2/L3.
- **Flaky activation для navigator-скилов** (выявлено в `session-ses_161a.md`, Bug 8). У L0 нет таблицы Key methods, и единственный устойчивый сигнал активации — это упоминание агентом имени скила в ответе («Per bsp-fundamentals: …»). Модель `9router/my-free` его произносит непредсказуемо. **Текущий обходной путь:**
  1. `test-harness.sh --runs 3` — majority vote по 3 прогонам;
  2. `--skill-type navigator` — включает 3-й путь активации по именам модулей из навигационной карты;
  3. в `## Smoke-test log` файла `reports/bsp-skills/<имя-скила>.md` фиксируется `T/N` (сколько прогонов из N активировались), а не bool.

  **Открытый вопрос:** достаточно ли этого для надёжного DoD-зачёта, или нужно добавить требование «обязательно встретить в ответе минимум 2 distinctive термина из `## Core concepts` скила» (например, `#Область`, `Служебный`, `СтандартныеПодсистемы` — слова, которые baseline-агент не произносит)? Решение принимается после прогона L1-скилов (где Key methods даёт устойчивый 2-й путь активации и проблема не должна стоять остро).

## 11.1. Триггеры ревизии скилов (Maintenance)

Скил может устаревать. Следующие события **обязывают** пересмотреть соответствующие скилы:

| Триггер | Что пересматривать | Кто отвечает |
|---|---|---|
| Выход новой версии БСП (3.1.12, 3.2.x и т. д.) | `metadata.version` всех скилов; Key methods на предмет переименований/удалённых методов | владелец репозитория |
| Smoke-test (§9 п.12) дал < 67% успешных прогонов | Patterns/Anti-patterns конкретного скила; добавить пояснения | автор скила |
| В репозитории появилась новая документация в `bsp3111_md/` (например, новая подсистема) | карта L0 `bsp-fundamentals`; возможно, новый скил в L2/L3 | владелец репозитория |
| Изменение конвенций/стиля скилов (обновление `agents-best-practices` или `prompt-crafting-guide`) | формат frontmatter и разделов | владелец репозитория |
| Обнаружена ошибка в Key methods (метод не существует / сигнатура неверна) | конкретный скил, `Verification log` в `reports/bsp-skills/<name>.md` | автор скила |
| Скил покрывает > 600 строк и не помещается в контекст | разбить на 2 скила (см. §2) | автор скила |
| 6+ месяцев с последней ревизии скила, активный код в репо | ручной re-audit | владелец репозитория |
| Обновление окружения (новая версия GNU grep / Git Bash / opencode) | прогнать `test-harness.sh --help`; запустить hello-world прогон с заведомо рабочим скилом и проверить, что метрики не схлопнулись в 0 (известный риск Bug 1: GNU grep 3.0 в Git Bash игнорирует кириллические диапазоны в `-E`-regex; см. `session-ses_161a.md`) | владелец репозитория |
| Появление новой модели в `opencode models` | прогнать L0/L1-скилы с `--runs 3` на новой модели; зафиксировать дельты в `## Smoke-test log` файла `reports/bsp-skills/<имя-скила>.md` | владелец репозитория |

**Процедура ревизии:** создать issue/задачу «Пересмотр скила `bsp-xxx`» → пробежаться по чек-листу §9 → закоммитить изменения с пометкой `[skill-revision]`.

## 14. Трекинг прогресса (чек-лист)

Параллельно с этим планом ведётся **краткий чек-лист** `plans/2026-06-06-bsp-skills-checklist.md` — по одной строке на скил с чек-боксом и кратким описанием назначения. Сгруппирован по фазам §8; L0/L1/L2/L3/audit.

### 14.1. Правило отметки чек-бокса (обязательное)

После того как скил прошёл **все три фазы §9** (RED baseline + GREEN чек-лист + REFACTOR), агент **обязан**:

1. Убедиться, что `reports/bsp-skills/<имя-скила>.md` существует и заполнен (см. §9.6): `## Verification log`, `## Self-check §9.1`, `## Baseline log`, `## Smoke-test log`. Без этого **не отмечать** чек-бокс.
2. Открыть `plans/2026-06-06-bsp-skills-checklist.md`.
3. Найти строку с именем скила.
4. Заменить `- [ ]` на `- [x]`.
5. Опционально — добавить дату публикации: `- [x] [2026-06-15]`.
6. Обновить «Прогресс-сводку» внизу чек-листа (колонка «Готово»).
7. **Закоммитить оба файла** в одном коммите: `SKILL.md` + `reports/bsp-skills/<имя>.md`.

> **Без отметки чек-бокса скил считается незавершённым**, даже если `SKILL.md` уже написан и прошёл §9. Это явный сигнал для пользователя и других агентов: «итерация не закрыта». Без заполненного `reports/bsp-skills/<имя>.md` чек-бокс **тоже** нельзя отмечать — см. §9.6 и §0.1 п.4.

### 14.2. Особые случаи

- **Скил признан ненужным по §9.0 RED-фазе** (baseline = 0 ошибок и расширение задач не помогло) → `- [-]` с обоснованием в `## Baseline log` файла `reports/bsp-skills/<имя-скила>.md`.
- **Скил в L3 — задачи нет** → оставить `- [ ]` и **не** отмечать до появления задачи; в прогресс-сводке он числится «в работе» условно.
- **Скил расщеплён** (например, `bsp-comms` → `bsp-comms-email-sms` + `bsp-comms-discussions`): исходный чек-бокс переходит в `- [x]` (заменён), новые добавляются в соответствующую фазу.

### 14.3. Синхронизация с большим планом

Чек-лист — **краткий снимок состояния**; большой план — **источник истины** по дизайну. Если в большом плане меняется § (например, переименование скила или расщепление L2-скила), **сначала** править большой план, **потом** синхронизировать чек-лист (переименовать строку, добавить новые).
