# Рефакторинг BSP-скилов: 4 umbrella → 1 самодостаточный скил

## Context

В репозитории `E:\gitReps\1c-ssl-skills` 4 umbrella-скила (`bsp-core`, `bsp-data`,
`bsp-ui-forms`, `bsp-ops`) покрывают БСП 3.1.11. Проблемы (из
`NOTES-bsp-skill-refactor.md`):

1. **Лишний хоп роутинга.** 4 скила с decision-tree и cross-cluster routing ссылаются
   друг на друга. БСП — один продукт, одна версия, один `src/cf/`. Кластерное деление —
   артефакт организации, не техническая граница.
2. **Дублирование кода.** 4 Python-скрипта (~300 строк каждый, 1217 суммарно), 90%
   дубликат. Отличаются только списками `MODULE_PREFIXES` (9/15/14/17).
   `BSP_SUBSYSTEMS` объявлен, но **нигде не используется** (мёртвый код).
3. **References дублируют греп.** Текущий формат — карта методов «Метод/Сигнатура/
   Сервер-Клиент/Стабильность/Назначение/Пример» (29 файлов). Агент находит методы сам
   через `Grep` по `src/cf/`. Реальная ценность скила — не карта, а **знание как
   правильно применять** БСП: workflow-сценарии, нюансы, антипаттерны.

**Цель:** скил учит правильно применять БСП, опираясь на офиц. документацию
`bsp3111_md/` и исходники `src/cf/` **при авторинге**, но в выходе скила —
самодостаточное содержание без ссылок на внешнюю документацию. Скрипты — помощники.

**Критичное ограничение:** `bsp3111_md/` (3045 .md, плоско, без подкаталогов по
подсистемам) НЕ поставляется со скилами. В скиле **запрещены** ссылки на `bsp3111_md/`
и хардкод-номера строк `src/cf/...:NNN`. Параметры и примеры — inline в references.
Ссылка на исходник — только `модуль + метод` (агент находит строку через `bsp_api.py`
или греп в своём `src/cf/`).

## Авторинг references: источник и метод поиска

`bsp3111_md/` — **источник при авторинге**, не часть скила. Структура: 3045 .md
плоско, без подкаталогов по подсистемам. Группировка — **по нумерации префикса**:
файл-описание подсистемы `NNNN_<транслит_подсистемы>.md` (напр.
`021_базовая_функциональность.md`) → подряд следующие файлы-методы этой подсистемы
(`022_…`, `023_…`) до следующего файла-описания подсистемы. 5 файлов-глав
(`001_…глава_1…` … `2970_глава_5…`).

**Метод поиска доки для подсистемы (Шаг 2):**

1. По платформённому имени подсистемы (из таблицы выше) построить транслит
   (`БазоваяФункциональность` → `базовая_функциональность`).
2. Найти файл-описание: `Glob bsp3111_md/*<транслит_подсистемы>.md`.
3. Читать подряд следующие файлы-методы до следующего файла-описания подсистемы.

**Метод поиска доки для конкретного метода:**

- `Glob bsp3111_md/*<транслит_метода>.md` (напр. `*сообщитьпользователю.md`).

**Dev-индекс (один раз, не в скил):** построить таблицу
`транслит_метода → NNNN_файл.md` грепом по именам `bsp3111_md/*.md`, чтобы быстро
находить дока-файл по имени метода из `src/cf/`. Это рабочий артефакт авторинга,
в скил не входит.

## Шаблон SKILL.md

Frontmatter:

```yaml
---
name: bsp
description: "Применение 1С:БСП 3.1.11 в прикладной разработке: правильные методы общих модулей, фоновые задания, обмен, ЭП/МЧД, печать, файлы, доступ, мониторинг. Самодостаточен без внешней документации."
---
```

Тело:

1. Краткое intro: чему учит, самодостаточность.
2. **Навигация по задачам** — таблица ~24 строк: «задача/ключевые слова →
   `references/<файл>.md` → подсистема БСП». Без decision-tree, без cross-cluster.
3. **Греп-шаблоны** для простых запросов:
   - экспортные методы модуля:
     `grep -Pn "^(Функция|Процедура)\s+\w+.*Экспорт" src/cf/CommonModules/<Модуль>/Ext/Module.bsl`
   - метод по имени во всех модулях:
     `grep -rPl "^(Функция|Процедура)\s+<Метод>\b" src/cf/CommonModules/`
   - регион метода (проверить stable/служебный):
     `grep -nP "^#Область\s+(ПрограммныйИнтерфейс|СлужебныйПрограммныйИнтерфейс|СлужебныеПроцедурыИФункции|УстаревшиеПроцедурыИФункции|Переопределение)" src/cf/CommonModules/<Модуль>/Ext/Module.bsl`
   - подсистема, содержащая модуль (по текстовым ссылкам в Subsystems XML):
     `grep -rl "CommonModule\.<Модуль>" src/cf/Subsystems/`
   - **не обещать** «найти подсистему объекта (Catalog/Document)» — это требует
     XML-разбора `Subsystems/<Имя>.xml` (`<Item>` со ссылками
     `Catalog./Document./CommonModule.`), тривиальный греп не покрывает; в SKILL.md
     указать как ручной шаг при необходимости.
4. **Скрипт `bsp_api.py`** — 3 команды, `--src <path>` обязательный:
   - `method <Имя>` — найти модуль по имени метода, вывести сигнатуру + имя
     региона + doc-комментарий + путь. Поиск по всем модулям.
   - `module <Имя>` — **все** экспортные методы модуля (любого региона:
     ПрограммныйИнтерфейс/СлужебныйПрограммныйИнтерфейс/УстаревшиеПроцедурыИФункции/
     Переопределение) с полными сигнатурами + регион. Агент сам классифицирует по
     региону.
   - `modules` — **только** модули со stable API (регион `ПрограммныйИнтерфейс` +
     экспорт). Фильтр по региону.
   - `modules-by-subsystem` и `detect` убраны (греп справляется; `BSP_SUBSYSTEMS`
     был мёртвым кодом). Разница `module` (один модуль, все регионы) vs `modules`
     (список модулей, только stable) — ключевая, не путать.

Объём ~140-160 строк. SKILL.md — **навигатор**: в нём нет backtick `Module.Method`
(они в references). Это влияет на верификацию — см. Шаг 5.

## Шаблон workflow-сценария в references

Эталон формата — **только `base-common.md`** (не `fundamentals.md` — тот навигатор,
см. ниже). Каждый файл:

- `# <Человеческое название>` — 1-3 строки что покрывает. БЕЗ ссылок на `bsp3111_md`.
- `## Модули` — краткий список с суффиксной логикой + ⚠️ несуществующие модули.
- `## Сценарии` — 4-10 workflow-сценариев.
- опц. `## Редкие методы` — влитые из `-key-methods.md`.

Один сценарий (~25-40 строк):

```markdown
### <Глагол задачи>

**Задача:** <одной строкой>

**Функция:**
`<Модуль>.<Метод>(<полная сигнатура>) Экспорт`
— <Процедура/Функция>, регион: `#Область ПрограммныйИнтерфейс` (stable)

**Параметры:**
- `<Имя>` (<Тип>) — <назначение, дефолт, нюансы>
...

**Пример:**
```bsl
<рабочий код вызова 3-8 строк с комментариями>
```

**Нюансы / антипаттерны:**
- ❌ <типичная ошибка> — <почему / чем заменить>
- <нюанс поведения (фоновое задание, транзакция, RLS и т. п.)>
```

**Маркировка регионов в сценариях** (критично — скрипт отдаёт имя региона, агент
классифицирует):

- `#Область ПрограммныйИнтерфейс` → stable, метка не нужна (по умолчанию).
- `#Область СлужебныйПрограммныйИнтерфейс` / `СлужебныеПроцедурыИФункции` → ⚠️
  служебный, в «Нюансах» указать стабильную альтернативу.
- `#Область УстаревшиеПроцедурыИФункции` → ⚠️ **deprecated**: «устаревший, не
  использовать в новом коде; альтернатива — `<Метод>`».
- `#Область Переопределение` → **переопределяемый хук**: БСП вызывает, прикладной
  код реализует. В сценариях помечать «хук переопределения — не вызывается из
  прикладного кода, а реализуется в прикладном модуле-переопределителе». Не давать как
  «вызови `Module.Method(...)`».

Эталон — `СообщитьПользователю` в `base-common.md`: сигнатура, 5 параметров inline,
пример с `Попытка/Исключение`, 4 антипаттерна (включая `Сообщить()` и несуществующий
`ОбщегоНазначения.ТекущаяДатаСеанса`).

## Скрипт bsp_api.py (~150 строк)

База: `.claude/skills/bsp-core/scripts/bsp_core_search.py`. Переиспользовать
`find_src_or_exit` дословно.

**Изменения `parse_export_methods`:**

- **Критично:** текущий код возвращает `region` как сводный тег
  (`"stable"`/`"unstable"`/`"override"`/`None`) — это **не имя области**. Для нового
  формата класть в стек **имя области**
  (`ПрограммныйИнтерфейс`/`СлужебныйПрограммныйИнтерфейс`/`СлужебныеПроцедурыИФункции`/`УстаревшиеПроцедурыИФункции`/`Переопределение`).
- Возвращать кортеж `(method_name, region_name, sig_text, doc_lines)`.
- `only_stable` параметр убрать; фильтрацию по региону делать **точечно** в команде
  `modules` (см. ниже), не в парсере.

**Сбор doc-комментариев (`doc_lines`) — регламентирован под объём:**
Реальный doc-блок БСП многострочный: над `СообщитьПользователю` — **51 строка**
(секции `// Параметры:`, `// Пример:`, «Случаи некорректного использования»), пустых
строк между doc и объявлением нет. Вывод полного doc раздул бы `method`.
Правила:

- Собирать строки `//` непосредственно над объявлением, **пропуская** пустые строки
  между doc-блоком и `Функция`/`Процедура` (если есть).
- **Обрезать**: выводить строки doc **до первой строки `// Пример:` (не включая
  её)** ИЛИ первые **15 строк** doc — что меньше. Секцию `// Пример:` не выводить
  (свой пример в сценарии references). Если `// Пример:` нет — первые 15 строк.
- Пропускать шапку-копирайт (строки `// Copyright…` в начале файла) — она не над
  методом.
- Формат вывода: сырые `//` строки, без рендеринга.

**Команды (3):**

- `method <Имя> --src <path>` — модуль + сигнатура + **имя региона** + doc-комментарий
  (обрезанный) + путь (без номера строки). Поиск по всем модулям, early-break на
  первом совпадении (уже есть). БЕЗ stable/unstable тега.
- `module <Имя> --src <path>` — **все** экспортные методы с полными сигнатурами +
  имя региона (агент классифицирует).
- `modules --src <path>` — модули со stable API: фильтр **`region_name == "ПрограммныйИнтерфейс"`** + хотя бы один экспортный метод. (Фильтр по региону
  остаётся — только переименован со сводного тега на имя области.)

**Удалить:** `MODULE_PREFIXES` (4 списка), `BSP_SUBSYSTEMS` (мёртвый код),
`is_relevant_module`, `cmd_modules_by_subsystem`, `cmd_detect`.

**Производительность (noted, опционально):** `method`/`modules` парсят все 801 модуль
без кеша — единичный вызов ~секунды-десятки секунд (приемлемо для инструмента
авторинга). При массовом авторинге/прогонах — опциональный dev-кеш
`.cache/modules_index.json` (имя модуля → список методов с регионами), строится один
раз. В скил не входит. Не блокер.

Формат вывода `method` (для `ПрограммныйИнтерфейс`):

```
Module: ОбщегоНазначения
Region: #Область ПрограммныйИнтерфейс
Path:   src/cf/CommonModules/ОбщегоНазначения/Ext/Module.bsl

Signature:
Процедура СообщитьПользователю(...) Экспорт

Doc-comment (до // Пример: / перв. 15 строк):
// Формирует и выводит сообщение...
// Параметры:
//   ТекстСообщенияПользователю - Строка - текст сообщения.
//   ...
```

Формат вывода `method` (для `Переопределение` — хук, не вызов):

```
Module: ОбщегоНазначенияПереопределяемый
Region: #Область Переопределение
Path:   src/cf/CommonModules/ОбщегоНазначенияПереопределяемый/Ext/Module.bsl

Signature:
Процедура ПриДобавленииПодсистемы(Знач НаименованиеПодсистемы) Экспорт

⚠️ Region: Переопределение — это хук переопределения.
БСП вызывает этот метод, прикладной код РЕАЛИЗУЕТ его в модуле-переопределителе.
НЕ вызывается из прикладного кода напрямую.

Doc-comment (до // Пример: / перв. 15 строк):
// Вызывается БСП при добавлении подсистемы...
```

Скрипт помечает `⚠️ Region: Переопределение` в выводе `method` для этого региона,
чтобы агент не выдал его как «вызови из прикладного». Аналогично — `⚠️ Region: УстаревшиеПроцедурыИФункции (deprecated)` и `⚠️ Region: СлужебныйПрограммныйИнтерфейс (обратная совместимость не гарантируется)`.

**Ручная верификация (5 задач, со скилом и без):** прогнать агентом набор задач ниже

- `method_accuracy`: методы в ответе реально существуют в `src/cf/` — сверка через
  греп-шаблон из SKILL.md (быстрее, чем `bsp_api.py method`):
  `grep -rPl "^(Функция|Процедура)\s+<Метод>\b" src/cf/CommonModules/` — найти модуль,
  затем `bsp_api.py method <Имя> --src src/cf` для сигнатуры/региона при сомнениях.
- `hallucinated = 0`: нет выдуманных `Module.Method`.
- `anti_pattern_violations = 0`: ответы не содержат ❌-паттернов из references
  (`Сообщить()` вместо `СообщитьПользователю`, `ОбщегоНазначения.ТекущаяДатаСеанса()`,
  прямой доступ к `БезопасноеХранилище` и т. п.).
- `skill_activated`: GREEN-ответы опираются на сценарии references (упомянутые
  функции совпадают с `base-common.md`/`longs-and-jobs.md`/…); RED — нет.

**Набор задач** (покрывают разные подсистемы):

1. «Выведи сообщение об ошибке с привязкой к реквизиту `Объект.Ответственный` и
   прерви операцию.» → `base-common.md`
2. «Запусти тяжёлую функцию `МойРасчёт()` в фоновом задании, верни прогресс клиенту.»
   → `longs-and-jobs.md`
3. «Сохрани пароль внешнего API-сервиса для учётной записи безопасно.» →
   `base-common.md`
4. «Подпиши документ электронной подписью и проверь подпись.» → `esign-mcd.md`
5. «Зарегистрируй изменения объекта для узла обмена.» → `data-exchange.md`

**Условие успеха (ручное):** по каждой задаче GREEN лучше или равен RED по

проверка самодостаточности: скил работает без внешней доки).

## Риски / открытые моменты

- **Объём авторинга.** ~24 файла × workflow-формат — большой шаг. Старт с
  `base-common.md` (эталон), согласовать формат, потом масштабировать.
- **Верификация (Шаг 5b).** По умолчанию ручная (5 задач × RED/GREEN с ручной
  сверкой `method_accuracy`/`hallucinated`/`anti_pattern`). Автоматизация через
  `test-harness.sh` — опциональная отдельная задача по `opencode-runner` (3-5ч +
  регресс-риск), не часть BSP-рефакторинга.
- **Гранула reference-файлов.** Полностью плоско по подсистемам — но часть связана
  (`base-common.md` = 3 модуля). Если файл >350 строк — выделить подсистему.
- **Служебные/устаревшие/переопределяемые методы.** Регион отдаётся именем, агент
  классифицирует. В сценариях метить ⚠️ (служебный), ⚠️ deprecated (устаревший),
  «хук переопределения» (Переопределение — не вызов, а реализация).
- **Производительность скрипта.** 801 модуль без кеша — опциональный dev-кеш при
  массовом авторинге.
