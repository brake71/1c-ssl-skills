# BSP Data Exchange (ОбменДанными + ОбменДаннымиВМоделиСервиса + EnterpriseData)

Скил по подсистемам обмена данными в БСП 3.1.11:

1. **`ОбменДанными`** — узлы планов обмена, правила регистрации изменений, сценарии синхронизации, обработчики коллизий, базовая РИБ (распределённая информационная база).
2. **`ОбменДаннымиВМоделиСервиса`** — обмен между областями данных в модели сервиса (SaaS): вложена в `РаботаВМоделиСервиса` и использует её механизмы.
3. **`СинхронизацияДанныхЧерезУниверсальныйФормат`** (EnterpriseData) — обмен через XML-сообщения универсального формата; реализован поверх `ОбменДаннымиXDTOСервер` (9745 строк) и пары `ТрансляцияXDTO*` (трансляция между версиями формата).

> ⚠️ **Главный тезис.** В БСП 3.1.11 у подсистем обмена **нет стабильного публичного API для прямого вызова** из прикладного кода, сопоставимого с `УправлениеПечатью.СформироватьПечатныеФормы`. Большинство общих модулей содержат `ПрограммныйИнтерфейс` (стабильный) с 1–2 методами, остальное уходит в `СлужебныйПрограммныйИнтерфейс` и `СлужебныеПроцедурыИФункции`. Правильный путь — **платформенный** `ПланыОбмена[ИмяПлана].*` плюс **переопределение** модулей `*Переопределяемый` и `*События`. Из-за этого скил отнесён к **L2** (понижен с L1): переопределяйте, не вызывайте.

> 🔗 **Связь с `bsp-longs-and-jobs`:** сценарии обмена большого объёма (`ВыполнитьОбменДаннымиПоСценариюОбменаДанными` в `ОбменДаннымиВызовСервера` и `ВыполнитьОбменДаннымиПоРегламентномуЗаданию` в `ОбменДаннымиСервер.СлужебныйПрограммныйИнтерфейс`) синхронно дёргать с управляемой формы нельзя — оборачивайте в `ДлительныеОперации.ВыполнитьФункцию`.

## When to use

- Программно зарегистрировать изменения объекта для узла плана обмена — `ПланыОбмена[ИмяПлана].ЗарегистрироватьИзменения(Узел, Объект)` (платформенный API; БСП работает поверх).
- Программно снять регистрацию изменений — `ПланыОбмена[ИмяПлана].УдалитьРегистрациюИзменений(Узел, Данные)`.
- Подключить новый план обмена к подсистеме обмена данными — в `ОбменДаннымиПереопределяемый.ПолучитьПланыОбмена` добавить `ПланыОбменаПодсистемы.Добавить(Метаданные.ПланыОбмена.МойПлан)`.
- Переопределить логику разрешения коллизий (главный vs подчинённый узел) — реализовать тело `ОбменДаннымиПереопределяемый.ПриКоллизииИзмененийДанных(ЭлементДанных, ПолучениеЭлемента, Отправитель, ПолучениеОтГлавного)`.
- Объявить поддерживаемые версии универсального формата EnterpriseData — реализовать тело `ОбменДаннымиПереопределяемый.ПриПолученииДоступныхВерсийФормата(ВерсииФормата)`.
- Зарегистрировать собственный обработчик трансляции между версиями EnterpriseData (1.6 → 1.7 и т. п.) — реализовать тело `ТрансляцияXDTOПереопределяемый.ЗаполнитьОбработчикиТрансляцииСообщений(МассивОбработчиков)`.
- Проверить, завершена ли настройка синхронизации для узла, — `ОбменДаннымиСервер.НастройкаСинхронизацииЗавершена(УзелОбмена)`.
- Найти узел плана обмена по его коду в конкретном плане — `ОбменДаннымиСервер.УзелПланаОбменаПоКоду(ИмяПланаОбмена, КодУзла)`.
- Узнать, что сеанс запущен в автономном рабочем месте (АРМ РИБ), — `ОбменДаннымиСервер.ЭтоАвтономноеРабочееМесто()`.
- Получить версию формата корреспондента для обмена — `ОбменДаннымиСервер.ВерсияКорреспондента(Корреспондент)`.

## Не использовать, если

- Нужно сформировать печатную форму или отчёт (табличный документ, варианты отчётов) — переключайтесь на `bsp-print-reports`. Там API `УправлениеПечатью.*` и `ВариантыОтчетов.*`.
- Нужна инфраструктура подключаемых команд (общая, не для печати) — `bsp-commands-external`. Сюда не входит: «обмен через файл», «отправить по e-mail», «выгрузить в Excel» — это команды, не обмен данными.
- Нужен запуск обмена в фоне (`ДлительныеОперации.ВыполнитьФункцию` + `ОжидатьЗавершение`) — `bsp-longs-and-jobs`. Сценарии обмена в `ОбменДаннымиВызовСервера.ВыполнитьОбменДаннымиПоСценариюОбменаДанными` не предназначены для прямого синхронного вызова с формы.
- Нужна базовая инфраструктура (сообщения пользователю, безопасное хранилище, общие утилиты) — `bsp-base-common`.
- Нужна работа с вложенными файлами или почтой (BSP `РаботаСФайлами`, `ЭлектроннаяПодпись`) — это отдельные подсистемы; интеграция с обменом там минимальна (вложения в сообщения обмена — это `СообщенияОбменаДанными*`, не путать с `ЭлектроннаяПочта`).

## Core concepts

### Главный тезис: платформенный API + переопределение событийных модулей

Подсистемы обмена данными устроены иначе, чем `Печать` или `ВариантыОтчетов`: они не предлагают разработчику «набор сервисных функций для вызова». Вместо этого предлагается:

1. **Платформенный слой** `ПланыОбмена[ИмяПлана].*` (`ЗарегистрироватьИзменения`, `УдалитьРегистрациюИзменений`, `ВыбратьИзменения`, `НайтиПоКоду`, `ЗаписатьСообщениеСИзменениями` и т. д.) — это ядро регистрации и транспорта. БСП поверх него добавляет правила, события и сценарии, но **не оборачивает** платформенный API в свой «стабильный» интерфейс.
2. **Событийные модули** `ОбменДаннымиСобытия` (подписки на события записи/удаления) — `МеханизмРегистрацииОбъектовПередЗаписью(ИмяПланаОбмена, Источник, Отказ)`, `МеханизмРегистрацииОбъектовПередУдалением(ИмяПланаОбмена, Источник, Отказ)`, `ЗарегистрироватьИзмененияДанных(Получатель, Данные, ПроверятьРазрешениеВыгрузки)`. **Не вызывайте** — БСП подключает их через подписки на события автоматически.
3. **Переопределяемые модули** `ОбменДаннымиПереопределяемый`, `ТрансляцияXDTOПереопределяемый` — `Экспорт`-методы с пустым телом по умолчанию. В них реализуют прикладную логику (состав планов обмена, правила коллизий, доступные версии формата). **Не вызывайте** — переопределяйте.
4. **Стабильный серверный API** — небольшой набор функций в `ОбменДаннымиСервер` (`УзлыОбменаБСП`, `УзелПланаОбменаПоКоду`, `ЭтоАвтономноеРабочееМесто`, `ВерсияКорреспондента`, `НастройкаСинхронизацииЗавершена`, `ЗавершитьНастройкуСинхронизацииДанных`, `ПрефиксИнформационнойБазы`). Реально полезных — 5–7 штук.
5. **Служебный API** — большая часть `ОбменДаннымиСервер` и почти весь `ОбменДаннымиСлужебный`. ⚠️ Обратная совместимость не гарантируется; используйте, только если стабильного аналога нет (например, `СоставПланаОбмена` в `ОбменДаннымиСлужебный`).

### Стабильный vs служебный vs переопределяемый vs событийный

- **Стабильный API** — `#Область ПрограммныйИнтерфейс` в `ОбменДаннымиСервер`, `ОбменДаннымиВызовСервера`. Этот набор **мал** (5–7 методов), в основном — справочные функции и переключатели состояния.
- **Стабильный для других подсистем БСП** — `#Область ДляВызоваИзДругихПодсистем` и `#Область СлужебныйПрограммныйИнтерфейс`. БСП их **рекомендует** вызывать из прикладного кода в отдельных случаях (например, `УзелПланаОбменаПоКоду` явно отмечен как рекомендуемая замена устаревшего метода в `ОбменДаннымиПовтИсп`), но формально это **не** публичный API.
- **Переопределяемый API** — `#Область ПрограммныйИнтерфейс` в модулях `*Переопределяемый` (`ОбменДаннымиПереопределяемый`, `ТрансляцияXDTOПереопределяемый`). Экспортные методы с пустым телом по умолчанию. **Не вызываются** — в них реализуют свою логику.
- **Событийный API** — `#Область ПрограммныйИнтерфейс` в `ОбменДаннымиСобытия`. Экспортные методы, к которым БСП подключается через подписки на события (`МеханизмРегистрацииОбъектовПередЗаписьюДокумента` и т. п.). **Не вызывайте** — изменение поведения делается через правила регистрации.
- **Служебный API** — `#Область СлужебныйПрограммныйИнтерфейс`, `#Область СлужебныеПроцедурыИФункции`, `#Область УстаревшиеПроцедурыИФункции`. ⚠️ Обратная совместимость не гарантируется.

### Подсистемы в дереве метаданных

- `ОбменДанными` — корневая. Содержит вложенные:
  - `ОбменДанными.Регистрация` — правила регистрации объектов, подписки на события;
  - `ОбменДанными.ТранспортСообщенийОбмена` — каналы доставки (файл, FTP, веб-сервис, e-mail).
- `ОбменДаннымиВМоделиСервиса` — вложена в `СтандартныеПодсистемы → РаботаВМоделиСервиса`. Использует инфраструктуру модели сервиса (`УправлениеДоступом`, `ОчередьЗаданий`, `РегламентныеЗадания`).
- `СинхронизацияДанныхЧерезУниверсальныйФормат` — отдельная корневая подсистема с **XDTO-пакетами** (`ExchangeMessages`, `EnterpriseData_*`), планами обмена формата `СинхронизацияДанныхЧерезУниверсальныйФормат`, обработчиками `ОбменДаннымиXDTOСервер`.

## Key methods

## Patterns

### Pattern 1: Регистрация изменений объекта программно через платформенный API

Используйте платформенный `ПланыОбмена[ИмяПлана].ЗарегистрироватьИзменения(Узел, Данные)`. БСП работает поверх платформенного механизма, а не «оборачивает» его:

```bsl
// ОбменДаннымиСервер (любой серверный модуль)
Процедура ЗарегистрироватьИзмененияДокумента(УзелОбмена, ДокументОбъект) Экспорт
    ИмяПланаОбмена = ОбменДаннымиПовтИсп.ИмяПланаОбмена(УзелОбмена);
    ПланыОбмена[ИмяПланаОбмена].ЗарегистрироватьИзменения(УзелОбмена, ДокументОбъект);
КонецПроцедуры
```

### Pattern 2: Подключение нового плана обмена к подсистеме

В `ОбменДаннымиПереопределяемый.ПолучитьПланыОбмена` добавьте свой план — БСП подхватит его для сценариев синхронизации, правил регистрации, UI-панели:

```bsl
// ОбменДаннымиПереопределяемый
Процедура ПолучитьПланыОбмена(ПланыОбменаПодсистемы) Экспорт
    ПланыОбменаПодсистемы.Добавить(Метаданные.ПланыОбмена._ДемоОбменВРаспределеннойИнформационнойБазе);
КонецПроцедуры
```

### Pattern 3: Обработка коллизий при обмене

В `ОбменДаннымиПереопределяемый.ПриКоллизииИзмененийДанных` решите, чья версия побеждает (главного или подчинённого узла), с учётом конкретного типа объекта:

```bsl
// ОбменДаннымиПереопределяемый
Процедура ПриКоллизииИзмененийДанных(Знач ЭлементДанных, ПолучениеЭлемента, Знач Отправитель, ПолучениеОтГлавного) Экспорт
    Если ТипЗнч(ЭлементДанных) = Тип("СправочникОбъект.Контрагенты") Тогда
        // В распределёнке подчинённый узел имеет приоритет
        ПолучениеЭлемента = ?(ПолучениеОтГлавного,
            ПолучениеЭлементаДанных.Игнорировать,
            ПолучениеЭлементаДанных.Принять);
    КонецЕсли;
КонецПроцедуры
```

### Pattern 4: Регистрация обработчика трансляции для новой версии EnterpriseData

При выпуске конфигурации, рассчитанной на новую версию универсального формата, зарегистрируйте модуль-транслятор в `ТрансляцияXDTOПереопределяемый`:

```bsl
// ТрансляцияXDTOПереопределяемый
Процедура ЗаполнитьОбработчикиТрансляцииСообщений(МассивОбработчиков) Экспорт
    МассивОбработчиков.Добавить(МойМодульТрансляцииEnterpriseData_1_6_в_1_7);
КонецПроцедуры
```

## Anti-patterns

1. ❌ Регистрировать изменения, вручную записывая в таблицы `РегистрацияИзмененийПланаОбмена` или через `НаборЗаписей`. Это обходит **платформенный** механизм, ломает авторегистрацию и приводит к расхождениям. ✅ Использовать `ПланыОбмена[ИмяПлана].ЗарегистрироватьИзменения(Узел, Данные)`.

2. ❌ Запускать тяжёлый сценарий обмена (`ОбменДаннымиВызовСервера.ВыполнитьОбменДаннымиПоСценариюОбменаДанными`) **синхронно** с управляемой формы. UI зависнет, пользователь не сможет прервать операцию. ✅ Оборачивать в `ДлительныеОперации.ВыполнитьФункцию` (см. `bsp-longs-and-jobs`) и показывать прогресс через `ДлительныеОперацииКлиент.ОжидатьЗавершение`.

3. ❌ Парсить XML сообщения обмена EnterpriseData вручную через `ЧтениеXML` / `ЗаписьXML`. Это **отключает** всю инфраструктуру правил конвертации, XDTO-пакеты и трансляции между версиями. ✅ Реализовать правила конвертации объектов в `ОбменДаннымиXDTOСервер`-стиле (через `КоллекцияПравилКонвертации`) и зарегистрировать обработчики трансляции версий в `ТрансляцияXDTOПереопределяемый.ЗаполнитьОбработчикиТрансляцииСообщений`.

4. ❌ Вызывать `ОбменДаннымиПереопределяемый.ПриКоллизииИзмененийДанных(ЭлементДанных, …)` из прикладного кода «чтобы задать правило». Метод **переопределяемый** — БСП его вызывает сам, ваш вызов будет лишним и не сработает (тело пустое, если не переопределено). ✅ Реализовать логику **в теле** `ПриКоллизииИзмененийДанных` в модуле `ОбменДаннымиПереопределяемый`.

5. ❌ Использовать `ОбменДаннымиПовтИсп.ЭтоАвтономноеРабочееМесто` или `ОбменДаннымиПовтИсп.НайтиУзелПланаОбменаПоКоду`. Оба помечены `// Устарела. Следует использовать ОбменДаннымиСервер.ЭтоАвтономноеРабочееМесто` / `ОбменДаннымиСервер.УзелПланаОбменаПоКоду`. ✅ Использовать методы в `ОбменДаннымиСервер`.

6. ❌ Использовать `ОбменДаннымиСлужебный.*` API как стабильный, потому что модуль — «тоже часть БСП». ⚠️ Служебный API не гарантирует обратной совместимости. ✅ Проверить, нет ли стабильного аналога в `ОбменДаннымиСервер` / `ОбменДаннымиВызовСервера`; если нет — зафиксировать зависимость в коде комментарием с указанием версии БСП.

7. ❌ Пытаться добавить подписку на событие обмена через `УстановитьОбработчик` или редактировать `ОбменДаннымиСобытия.ЗарегистрироватьИзмененияДанных` напрямую. Это сломает работу в режиме совместимости расширений. ✅ Использовать механизм правил регистрации объектов плана обмена + `ОбменДаннымиПереопределяемый.ПриКоллизииИзмененийДанных` для тонкой настройки.

8. ❌ Считать, что `ПередЗаписьюДокумента` / `ПриЗаписиДокумента` в `ОбменДаннымиСобытия` — это стабильный API, который можно вызывать. Эти методы — **событийные**, БСП подключает их через подписки на события (`МеханизмРегистрацииОбъектовПередЗаписьюДокумента`). Изменяйте правила регистрации объекта в плане обмена, а не вызывайте обработчики напрямую.

## How to explore deeper

### Где искать модули в выгрузке конфигурации

В целевом репозитории агента исходники БСП **уже включены** как подмножество конфигурации. Найди общий модуль и открой его BSL/XML:

- `ОбменДаннымиСервер` — `CommonModules/ОбменДаннымиСервер/Ext/Module.bsl` (Сервер, ВнешнееСоединение; ~9527 строк).
- `ОбменДаннымиКлиент` — `CommonModules/ОбменДаннымиКлиент/Ext/Module.bsl` (Клиент).
- `ОбменДаннымиВызовСервера` — `CommonModules/ОбменДаннымиВызовСервера/Ext/Module.bsl` (Сервер; обёртки для клиентских вызовов).
- `ОбменДаннымиПереопределяемый` — `CommonModules/ОбменДаннымиПереопределяемый/Ext/Module.bsl` (Сервер; **переопределение**, не вызов).
- `ОбменДаннымиСобытия` — `CommonModules/ОбменДаннымиСобытия/Ext/Module.bsl` (Сервер; **событийные обработчики**, не вызов).
- `ОбменДаннымиСлужебный` — `CommonModules/ОбменДаннымиСлужебный/Ext/Module.bsl` (Сервер; ⚠️ служебный API).
- `ОбменДаннымиПовтИсп` — `CommonModules/ОбменДаннымиПовтИсп/Ext/Module.bsl` (Сервер, кэш; **ПрограммныйИнтерфейс содержит только устаревшие методы**).
- `ОбменДаннымиВМоделиСервиса` — `CommonModules/ОбменДаннымиВМоделиСервиса/Ext/Module.bsl` (Сервер; вложен в `РаботаВМоделиСервиса`).
- `ОбменДаннымиВМоделиСервисаПереопределяемый` — `CommonModules/ОбменДаннымиВМоделиСервисаПереопределяемый/Ext/Module.bsl` (Сервер; **переопределение**).
- `ОбменДаннымиXDTOСервер` — `CommonModules/ОбменДаннымиXDTOСервер/Ext/Module.bsl` (Сервер; ~9745 строк, правила конвертации EnterpriseData).
- `ТрансляцияXDTO` — `CommonModules/ТрансляцияXDTO/Ext/Module.bsl` (Сервер; ⚠️ служебный).
- `ТрансляцияXDTOПереопределяемый` — `CommonModules/ТрансляцияXDTOПереопределяемый/Ext/Module.bsl` (Сервер; **переопределение**).
- `ОбменДаннымиРегистрацияСервер` — `CommonModules/ОбменДаннымиРегистрацияСервер/Ext/Module.bsl` (Сервер; ⚠️ служебный: выборочная регистрация изменений).
- `ОбменДаннымиТрансляцияФорматаПовтИсп` — `CommonModules/ОбменДаннымиТрансляцияФорматаПовтИсп/Ext/Module.bsl` (Сервер; кэш правил трансляции).

Точки входа в подсистемы:
- `Subsystems/СтандартныеПодсистемы/Subsystems/ОбменДанными.xml` — корневая подсистема `ОбменДанными` (включает вложенные `Регистрация` и `ТранспортСообщенийОбмена`).
- `Subsystems/СтандартныеПодсистемы/Subsystems/РаботаВМоделиСервиса/Subsystems/ОбменДаннымиВМоделиСервиса.xml` — вложенная подсистема `ОбменДаннымиВМоделиСервиса` (SaaS).
- `Subsystems/СтандартныеПодсистемы/Subsystems/СинхронизацияДанныхЧерезУниверсальныйФормат.xml` — корневая подсистема `СинхронизацияДанныхЧерезУниверсальныйФормат` (EnterpriseData).

- `CommonModules/ТрансляцияXDTO*/Ext/Module.bsl` — семейство `ТрансляцияXDTO` (2 модуля; трансляция между версиями EnterpriseData).
- `CommonModules/_ДемоОбмен*/Ext/Module.bsl` — демонстрационные модули обмена (для примеров).
- `Subsystems/СтандартныеПодсистемы/Subsystems/ОбменДанными.xml` — точка входа подсистемы `ОбменДанными`.
- `Subsystems/СтандартныеПодсистемы/Subsystems/РаботаВМоделиСервиса/Subsystems/ОбменДаннымиВМоделиСервиса.xml` — точка входа подсистемы `ОбменДаннымиВМоделиСервиса`.
- `Subsystems/СтандартныеПодсистемы/Subsystems/СинхронизацияДанныхЧерезУниверсальныйФормат.xml` — точка входа подсистемы `СинхронизацияДанныхЧерезУниверсальныйФормат`.

### На что обратить внимание в дереве метаданных

- **`ПланыОбмена.*`** в конфигурации — ядро обмена. Каждый план имеет таблицу `РегистрацияИзменений` (платформенный механизм). Правила регистрации и авторегистрация настраиваются в свойствах плана обмена (`Состав`, флаг `Авторегистрация` для каждого типа).
- **`ОпределяемыеТипы.ПланОбменаОсновной` / `…Дополнительный`** — определяют, какие планы обмена участвуют в подсистеме обмена БСП.
- **`РегистрыСведений.ОбщиеНастройкиУзловИнформационныхБаз`** — состояние узлов РИБ (`НастройкаЗавершена`, `НачальныйОбразСоздан`); используется `ОбменДаннымиСервер.НастройкаСинхронизацииЗавершена`.
- **`РегистрыСведений.ПравилаДляОбменаДанными`** — сохранённые правила обмена (XML-файлы правил).
- **`Справочники.СценарииОбменовДанными`** — сценарии обмена (откуда, куда, по расписанию). Запускаются регламентным заданием `ОбменДанными.ВыполнениеОбменовДанными`.
- **`ПодпискиНаСобытия`** в подсистеме `ОбменДанными` — `МеханизмРегистрацииОбъектовПередЗаписью*`, `МеханизмРегистрацииОбъектовПередУдалением` и т. п. Они подключают `ОбменДаннымиСобытия.*` к событиям записи/удаления объектов.
- **`Обработки.ВыполнениеОбменовДанными`** / **`РегламентныеЗадания`** — `ОбменДанными.ВыполнениеОбменовДанными` (фоновое выполнение сценариев обмена).
- **`XDTO-пакеты`** `ExchangeMessages`, `EnterpriseData_*` — схемы универсального формата. В `ОбменДаннымиXDTOСервер` через `КоллекцияПравилКонвертации` описывается маппинг «наш объект ↔ XDTO-объект».
- **В демонстрационной конфигурации** — `_ДемоОбменДаннымиСобытия`, `_ДемоОбменМобильный*` (мобильный клиент): полезные примеры регистрации и подключения обменов.
