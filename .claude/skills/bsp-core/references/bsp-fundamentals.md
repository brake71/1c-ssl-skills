
# BSP Fundamentals (StandardSubsystems 3.1.11)

Навигационный скил по библиотеке стандартных подсистем 1С. Цель — помочь агенту правильно «видеть» БСП в репозитории конфигурации 1С: находить нужный общий модуль за 3 шага, отличать стабильный API от служебного, понимать систему суффиксов и не путаться в ~70 подсистемах верхнего уровня.

Скил не описывает конкретные методы (`Модуль.Метод(...)`). Это работа L1-скилов (`bsp-base-common`, `bsp-users-access`, `bsp-print-reports`, `bsp-commands-external`, `bsp-longs-and-jobs`). Здесь — только карта местности и правила навигации.

## When to use

- Нужно найти общий модуль, отвечающий за функциональность из конкретной подсистемы БСП («где реализована печать? адресный классификатор? регламентные задания?»).
- Нужно понять, что означает суффикс в имени общего модуля (`Клиент`, `Сервер`, `КлиентСервер`, `ВызовСервера`, `Глобальный`, `ПовтИсп`, `Служебный`, `Переопределяемый`) и какой из вариантов вызывать из прикладного кода.
- Нужно отличить стабильный программный интерфейс от служебного (`#Область ПрограммныйИнтерфейс` vs `#Область СлужебныйПрограммныйИнтерфейс` vs `#Область СлужебныеПроцедурыИФункции`).
- Нужно решить, к какой подсистеме БСП относится задача, чтобы выбрать правильный L1/L2 скил.
- Нужно проверить, существует ли определённый общий модуль (типовая ошибка — выдумывать модуль `ОбщегоНазначенияСлужебный` без суффиксов, которого в БСП 3.1.11 не существует).

## Не использовать, если

- Нужно вызвать конкретный метод БСП (например, `ОбщегоНазначения.ТекущаяДатаСеанса`, `Пользователи.ТекущийПользователь`) — переключайтесь на L1-скил по соответствующей подсистеме (`bsp-base-common`, `bsp-users-access` и т. д.).
- Задача не связана с БСП вообще (типовое расширение конфигурации без вызовов БСП-модулей).
- Нужно настроить подключение/конфигурирование БСП в конфигурации (это задача архитектора, не покрывается этим скилом).
- Нужно решить вопрос совместимости версий БСП (3.1.11 ↔ 3.2.x) — выходит за рамки скила.

## Core concepts

### Что такое БСП

**БСП** (Библиотека стандартных подсистем, корневая подсистема `СтандартныеПодсистемы`) — набор из ~70 переиспользуемых подсистем 1С, которые встраиваются в прикладные конфигурации как подмножество. Версия 3.1.11. БСП реализует кросс-конфигурационную функциональность: пользователи и права, печать и отчёты, обмен данными, файлы, электронная подпись, длительные операции, регламентные задания и т. д.

В целевом репозитории агента БСП **уже включена** в конфигурацию как подмножество — исходники модулей БСП физически присутствуют в виде BSL/XML-файлов. Документации БСП в репо нет, но есть код — поэтому при необходимости агент сверяется с реальными модулями, а не с абстрактной документацией.

### Где искать исходники в целевом репо

Каталог с выгрузкой конфигурации (cf-дерево) содержит стандартные подкаталоги объектов метаданных:

- `CommonModules/<ИмяМодуля>/Ext/Module.bsl` — исходный код общего модуля БСП.
- `CommonModules/<ИмяМодуля>/Ext/Module.xml` — метаданные модуля (Сервер/Клиент/ВнешнееСоединение, режим повторного использования).
- `Subsystems/СтандартныеПодсистемы/Subsystems/<ИмяПодсистемы>.xml` — описание подсистемы верхнего уровня.
- `InformationRegisters/<ИмяРегистра>/` — регистры сведений БСП (например, `БезопасноеХранилищеДанных`).
- `Catalogs/`, `Documents/`, `Reports/` — прикладные объекты БСП (справочники пользователей, ролей, и т. д.).

### Именование общих модулей — без префикса, с суффиксами

В отличие от других вендорских библиотек, общие модули БСП **не имеют единого префикса** (нет `БСП_…`, нет `СтПС_…`). Модуль называется по своей подсистеме напрямую: `ОбщегоНазначения`, `Пользователи`, `АдресныйКлассификатор`, `УправлениеПечатью`, `ДлительныеОперации`, `РегламентныеЗаданияСервер`.

Префикс `БСП_` **не используется** ни в одном модуле БСП. **Не путать** с подсистемой `БТС` (Базовые табличные структуры) и её модулями `БТС`, `БТСКлиент` — это отдельная подсистема, не «префикс БСП».

Разделение на варианты контекста и режима выполнения сделано через **суффиксы имени** модуля:

| Суффикс | Что означает | Пример |
|---|---|---|
| (без суффикса) | Серверный код | `ОбщегоНазначения`, `Пользователи`, `УправлениеПечатью` |
| `Клиент` | Клиентский код (тонкий / веб-клиент). Не вызывает сервер сам, ждёт контекста формы | `ОбщегоНазначенияКлиент`, `ПользователиКлиент` |
| `КлиентСервер` | Безопасный код — работает и на сервере, и на клиенте, без обращения к БД | `ОбщегоНазначенияКлиентСервер`, `СтроковыеФункцииКлиентСервер` |
| `ВызовСервера` | Клиентский модуль, который **делает серверный вызов** (помечен «Вызов сервера» в Конфигураторе) | `ОбщегоНазначенияВызовСервера`, `БизнесПроцессыИЗадачиВызовСервера` |
| `Глобальный` | Процедуры глобального контекста (вызываются по короткому имени без префикса модуля) | `ОбщегоНазначенияГлобальный`, `УправлениеПечатьюГлобальный` |
| `ПовтИсп` | Режим «Повторное использование возвращаемых значений» — кэширование | `ОбщегоНазначенияКлиентПовтИсп`, `АдресныйКлассификаторПовтИсп` |
| `Служебный` | Внутренний API. Экспортные методы есть, но **обратная совместимость не гарантируется**. Использовать только если нет стабильного аналога | `ОбщегоНазначенияСлужебныйКлиент`, `ПользователиСлужебный`, `РегламентныеЗаданияСлужебный` |
| `Переопределяемый` | Содержит «крючки» для переопределения поведения БСП в прикладной конфигурации. Эти методы **не вызывают**, а реализуют под свои нужды | `ОбщегоНазначенияПереопределяемый`, `ПодключаемыеКомандыПереопределяемый` |
| `Локализация`, `РФ`, `ВМоделиСервиса` | Региональные / контекстные варианты | `СтроковыеФункцииКлиентСерверЛокализация`, `УправлениеПечатьюРФ`, `ПользователиСлужебныйВМоделиСервиса` |

Суффиксы можно комбинировать: `ОбщегоНазначенияСлужебныйКлиент` = служебный + клиент; `ПользователиСлужебныйВМоделиСервисаКлиентСервер` = служебный + контекст модели сервиса + клиент-сервер.

### Типичные «ловушки» именования

В БСП 3.1.11 есть несколько имён, которые легко выдумать по аналогии — но их **не существует**:

- **`ОбщегоНазначенияСлужебный`** (без суффиксов `Клиент`/`КлиентСервер`) — **не существует**. Служебный вариант называется `ОбщегоНазначенияСлужебныйКлиент` или `ОбщегоНазначенияСлужебныйКлиентСервер`.
- **`ФайловаяСистемаКлиентСервер`** — **не существует**. Есть `ФайловаяСистема` (сервер), `ФайловаяСистемаКлиент` (клиент). Для общего контекста — `ФайловаяСистемаСлужебныйКлиентСервер` (служебный).
- **`ДлительныеОперацииСлужебный`** — **не существует**. Служебные функции «Длительных операций» встроены в основные модули `ДлительныеОперации`, `ДлительныеОперацииКлиент`, `ДлительныеОперацииВызовСервера`.
- **`РегламентныеЗадания`** (как общий модуль без суффикса) — **не существует**. Есть `РегламентныеЗаданияСервер`, `РегламентныеЗаданияКлиент`, `РегламентныеЗаданияПереопределяемый`, `РегламентныеЗаданияСлужебный`. Подсистема `РегламентныеЗадания` существует — это объект метаданных верхнего уровня, не общий модуль.
- **`БизнесПроцессыИЗадачи`** (как общий модуль без суффикса) — **не существует**. Серверный модуль — `БизнесПроцессыИЗадачиСервер` (с суффиксом).
- **`БезопасноеХранилище`** как общий модуль — **не существует**. Это **регистры сведений** `БезопасноеХранилищеДанных` и `БезопасноеХранилищеДанныхОбластейДанных`. Работа с ними — через методы серверного модуля `ОбщегоНазначения` (`ЗаписатьДанныеВБезопасноеХранилище`, `ПрочитатьДанныеИзБезопасногоХранилища`, `УдалитьДанныеИзБезопасногоХранилища`).
- **`МашиночитаемыеДоверенности`** как общий модуль — **не существует** (без суффикса ФНС). Есть `МашиночитаемыеДоверенностиФНС`, `МашиночитаемыеДоверенностиФНСКлиент` и т. д. — все с суффиксом `ФНС`.
- **`УправлениеДоступомКлиент`** — **не существует**. Клиентский вариант есть только в служебном слое: `УправлениеДоступомСлужебныйКлиент`. Серверный модуль — `УправлениеДоступом` (без суффикса), переопределяемый — `УправлениеДоступомПереопределяемый`.

Перед тем как сослаться на модуль в коде — проверьте его существование, поискав `CommonModules/<ИмяМодуля>/Ext/Module.bsl` в дереве конфигурации.

### Разделение на области в общем модуле БСП

Внутри одного общего модуля БСП код **обязательно** разделён на области через директиву препроцессора `#Область`. Это не комментарии (`//`), а валидные конструкции языка 1С, которые видны в Конфигураторе и парсятся инструментами.

Стандартный набор областей в общем модуле БСП:

| Область | Что внутри | Можно ли вызывать из прикладного кода |
|---|---|---|
| `#Область ПрограммныйИнтерфейс` | Стабильные экспортные функции и процедуры. Обратная совместимость поддерживается между минорными версиями БСП | **Да**. Это основной API подсистемы |
| `#Область Переопределение` | Экспортные методы, **предназначенные для замещения** в одноимённом переопределяемом модуле прикладной конфигурации | **Нет, только переопределять** в своём модуле `*Переопределяемый` |
| `#Область СлужебныйПрограммныйИнтерфейс` | Экспортные методы для внутренних нужд БСП. Используются другими подсистемами БСП. **Обратная совместимость не гарантируется** | **Только если нет стабильного аналога** и с явной отметкой ⚠️ «служебный API» в коде |
| `#Область СлужебныеПроцедурыИФункции` | Глубоко внутренние процедуры. Не экспортные, не должны вызываться извне | **Нет** |

Все region markers — `#Область` (директива препроцессора 1С). Закрывающая директива — `#КонецОбласти`. Использовать `//` как region marker — ошибка, такие комментарии не видны Конфигуратору как структурные блоки.

Пример заголовка процедуры из области программного интерфейса:

```bsl
#Область ПрограммныйИнтерфейс

// Возвращает значение системной даты с учётом часового пояса сеанса.
Функция ТекущаяДатаСеанса() Экспорт
    // ...
КонецФункции

#КонецОбласти
```

### Иерархия подсистем БСП 3.1.11 (~70 подсистем)

Подсистемы БСП лежат в дереве метаданных по адресу `Subsystems/СтандартныеПодсистемы/Subsystems/`. На верхнем уровне — около 70 файлов `.xml` (каждый описывает одну подсистему), плюс одноимённые каталоги для подсистем с вложенной структурой (`БазоваяФункциональность/`, `ОбменДанными/`, `Пользователи/` и т. п.). Внутри этих каталогов могут быть вложенные подсистемы 2-го уровня.

Для удобства навигации подсистемы делятся на 7 тематических групп (логическая группировка для агента; в `.xml`-структуре её нет):

| Группа | Подсистемы (имена объектов метаданных) |
|---|---|
| **Средства администрирования и обслуживания** | `Администрирование`, `ДополнительныеОтчетыИОбработки`, `КонтрольРаботыПользователей`, `НастройкиПрограммы`, `ОбновлениеКонфигурации`, `ОтчетОДвиженияхДокумента`, `ОценкаПроизводительности`, `ПоискИУдалениеДублей`, `ПрофилиБезопасности`, `РезервноеКопированиеИБ`, `УдалениеПомеченныхОбъектов`, `УправлениеИтогамиИАгрегатами` |
| **Администрирование пользователей и прав доступа** | `Пользователи` (+`ВМоделиСервиса`), `УправлениеДоступом` (+`ВМоделиСервиса`), `ЗавершениеРаботыПользователей` |
| **Интеграция с другими приложениями и системами** | `ВнешниеКомпоненты` (+`ВМоделиСервиса`), `ЗагрузкаДанныхИзФайла`, `ИнтерфейсOData`, `ОбменДанными` (+`ВМоделиСервиса`), `ОбращенияВТехническуюПоддержку`, `Обсуждения`, `ОтправкаSMS`, `ПолучениеФайловИзИнтернета`, `РаботаСПочтовымиСообщениями`, `РассылкаОтчетов`, `СервисМобильнойПодписи`, `ТранспортСообщенийОбмена`, `ЭлектроннаяПодписьСервисаDSS` |
| **Технологические механизмы и программные интерфейсы** | `БазоваяФункциональность` (+`ВМоделиСервиса`), `ГенерацияШтрихкода`, `ЗапретРедактированияРеквизитовОбъектов`, `Мультиязычность`, `ОбновлениеВерсииИБ` (+`ВМоделиСервиса`), `ПодключаемыеКоманды`, `ПрефиксацияОбъектов`, `РаботаВМоделиСервиса`, `ЦентрМониторинга` |
| **Прикладные подсистемы и рабочие места пользователей** | `Анкетирование`, `БизнесПроцессыИЗадачи`, `ВариантыОтчетов`, `Взаимодействия`, `РаботаСФайлами` (+`ВМоделиСервиса`), `ТекущиеДела`, `УчетОригиналовПервичныхДокументов`, `ШаблоныСообщений` |
| **Нормативно-справочная информация и классификаторы** | `АдресныйКлассификатор`, `Банки`, `Валюты`, `ГрафикиРаботы`, `КалендарныеГрафики`, `Организации` |
| **Сервисные подсистемы** | `ВерсионированиеОбъектов`, `ВыгрузкаОбъектовВФайлы`, `ГрупповоеИзменениеОбъектов`, `ДатыЗапретаИзменения`, `ЗаметкиПользователя`, `ЗащитаПерсональныхДанных`, `ИнформацияПриЗапуске`, `КонструкторФормул`, `КонтактнаяИнформация`, `КонтрольВеденияУчета`, `МашиночитаемыеДоверенности`, `НапоминанияПользователя`, `НастройкаПорядкаЭлементов`, `Печать`, `ПолнотекстовыйПоиск`, `ПроверкаЛегальностиПолученияОбновления`, `Регистрация`, `РегламентныеЗадания`, `Свойства`, `СклонениеПредставленийОбъектов`, `СтруктураПодчиненности`, `ПереводТекста`, `ЭлектроннаяПодпись` |

Отдельный «хвост»: подсистемы с префиксом `_Демо…` — демонстрационные, не для прода. Подсистемы `БТС` и `БТСКлиент` — базовые табличные структуры (отдельная подсистема).

### Карта «подсистема L1 → ключевые общие модули»

Для типовых задач прикладной разработки нужны эти семейства модулей. Это **полный** перечень модулей семейства, не сокращённый — агенту нужно знать все варианты, чтобы выбрать правильный по контексту вызова (Сервер / Клиент / КлиентСервер / ВызовСервера / Глобальный / ПовтИсп / Служебный / Переопределяемый).

**Базовая функциональность (общие утилиты, файлы, длительные операции, журнал регистрации):**
- `ОбщегоНазначения`, `ОбщегоНазначенияКлиент`, `ОбщегоНазначенияКлиентСервер`, `ОбщегоНазначенияВызовСервера`, `ОбщегоНазначенияГлобальный`, `ОбщегоНазначенияПереопределяемый`, `ОбщегоНазначенияКлиентПереопределяемый`, `ОбщегоНазначенияКлиентПовтИсп`
- Служебные: `ОбщегоНазначенияСлужебныйКлиент`, `ОбщегоНазначенияСлужебныйКлиентСервер` (⚠️ нет модуля `ОбщегоНазначенияСлужебный` без суффиксов)
- БТС-специфичные: `ОбщегоНазначенияБТС`, `ОбщегоНазначенияБТСПовтИсп`, `ОбщегоНазначенияВызовСервераБТС`
- Строки: `СтроковыеФункции`, `СтроковыеФункцииКлиент`, `СтроковыеФункцииКлиентСервер` (+ `Локализация`, `РФ`)
- Файлы: `ФайловаяСистема`, `ФайловаяСистемаКлиент`, `ФайловаяСистемаСлужебныйКлиент`, `ФайловаяСистемаСлужебныйКлиентСервер` (⚠️ нет `ФайловаяСистемаКлиентСервер`)
- Длительные операции: `ДлительныеОперации`, `ДлительныеОперацииКлиент`, `ДлительныеОперацииВызовСервера` (⚠️ нет `ДлительныеОперацииСлужебный`)
- Безопасное хранилище: **не общий модуль**, а регистр сведений `БезопасноеХранилищеДанных`; работа — через `ОбщегоНазначения.{Записать,Прочитать,Удалить}ДанныеВБезопасноеХранилище`

**Пользователи и управление доступом:**
- `Пользователи`, `ПользователиКлиент`, `ПользователиКлиентСервер`, `ПользователиПереопределяемый`
- Служебные: `ПользователиСлужебный`, `ПользователиСлужебныйКлиент`, `ПользователиСлужебныйКлиентСервер`, `ПользователиСлужебныйПовтИсп`, `ПользователиСервиса`

### «Тесная интеграция»

Подсистемы БСП **тесно связаны** между собой: одна подсистема вызывает другую, события одной обрабатываются другой. Прикладному разработчику это значит:

- Нельзя подключить «только одну» подсистему БСП без её зависимостей.
- При вызове любого метода БСП агенту следует ожидать, что внутри метод дёрнет другие модули БСП (логирование, проверка прав, замер производительности, фоновые задания).
- Подмена реализации одного метода БСП своей логикой ломает соседние подсистемы. Изменения вносить только через переопределяемые модули (`*Переопределяемый`).

## Patterns

### Pattern 1: Найти нужный общий модуль за 3 шага

Когда нужно сделать действие, связанное с подсистемой БСП (печать, обмен, пользователи, файлы):

1. **Определить подсистему** по карте групп выше (Сервисные / Прикладные / Технологические и т. д.). Например: «нужна печать» → группа Сервисные, подсистема `Печать`.
2. **Подобрать имя модуля** по подсистеме. Корневое имя модуля обычно совпадает с понятием функции внутри подсистемы — для `Печать` это `УправлениеПечатью`, для `АдресныйКлассификатор` — одноимённый модуль `АдресныйКлассификатор`, для `БизнесПроцессыИЗадачи` — `БизнесПроцессыИЗадачиСервер` (с суффиксом).
3. **Выбрать вариант суффикса** под контекст вызова:
   - Серверный код без UI → модуль без суффикса (или с суффиксом `Сервер`, если он есть в семействе)
   - Клиентский обработчик команды → `*Клиент` или `*ВызовСервера` (если нужно дёрнуть сервер)
   - Общий код в `КлиентСервер`-модуле → `*КлиентСервер`
   - Кэширование результата → `*ПовтИсп`
   - Переопределение поведения БСП → `*Переопределяемый`

После определения имени — **проверить существование модуля** в `CommonModules/` дерева конфигурации (особенно если имя похоже на «выдуманное» — см. раздел «Типичные ловушки именования»).

### Pattern 2: Отличить стабильный API от служебного

Открыв `CommonModules/<Имя>/Ext/Module.bsl`, ориентируйтесь по областям:

```bsl
#Область ПрограммныйИнтерфейс    // <- стабильный API, вызывайте свободно
Функция СделатьЧтоТо() Экспорт
    ...
КонецФункции
#КонецОбласти

#Область СлужебныйПрограммныйИнтерфейс    // <- ⚠️ служебный, без гарантий совместимости
Функция СлужебныйМетод() Экспорт
    ...
КонецФункции
#КонецОбласти

#Область СлужебныеПроцедурыИФункции    // <- внутреннее, не экспорт, не вызывать
Процедура ВнутренняяОбработка()
    ...
КонецПроцедуры
#КонецОбласти
```

При выборе метода для прикладного кода: предпочтение — методам из `#Область ПрограммныйИнтерфейс`. Если нужного метода там нет, и аналог есть только в `#Область СлужебныйПрограммныйИнтерфейс` — допустимо использовать с явной отметкой в коде:

```bsl
// ⚠️ Служебный API БСП — обратная совместимость не гарантируется.
// Альтернативы в стабильном API на момент N версии нет.
РегламентныеЗаданияСлужебный.ДобавитьЗадание(Расписание, ...);
```

### Pattern 3: Найти примеры использования внутри самой БСП

Лучший источник примеров API подсистемы — другие подсистемы БСП, которые её уже используют. Поиск:

- `grep -rln "ОбщегоНазначения\.СообщитьПользователю" CommonModules/` — все вызовы конкретного метода в других модулях БСП.
- `grep -rln "ОбщегоНазначения\." CommonModules/_Демо*/` — демо-модули БСП (каталоги с префиксом `_Демо…` в `CommonModules/`) специально показывают идиоматичные вызовы.
- `grep -rln "<имя метода>" Documents/*/Ext/ObjectModule.bsl` — где из прикладного кода типовой конфигурации вызывается этот метод.

### Pattern 4: Проверить существование модуля или метода перед использованием

Перед тем как написать `<Модуль>.<Метод>(...)` в скилла или коде:

```bash
# Существует ли модуль?
ls CommonModules/<ИмяМодуля>/Ext/Module.bsl

# Существует ли экспортный метод в этом модуле?
grep -nE "^(Функция|Процедура) <ИмяМетода>\(.*\) Экспорт" CommonModules/<ИмяМодуля>/Ext/Module.bsl
```

Если grep ничего не вернул — метод (или модуль) выдуман. Не упоминайте его в коде или документации, не подгоняйте имя под аналогию с другими подсистемами.

## Anti-patterns

- ❌ **Использовать префикс `БСП_…` в именах общих модулей.** Никакого префикса нет — модули БСП называются по своей подсистеме напрямую. Не путать с `БТС` (отдельная подсистема Базовые табличные структуры), которая выглядит как «префикс», но префиксом не является.
- ❌ **Выдумывать «логичные» имена модулей по аналогии.** Не существует `ОбщегоНазначенияСлужебный` (без суффиксов), `ФайловаяСистемаКлиентСервер`, `ДлительныеОперацииСлужебный`, `БизнесПроцессыИЗадачи` (без суффикса `Сервер`), `РегламентныеЗадания` (без суффикса), `УправлениеДоступомКлиент` (без префикса `Служебный`), `МашиночитаемыеДоверенности` (без суффикса `ФНС`), `БезопасноеХранилище` как общий модуль. Перед использованием — проверять `CommonModules/<Имя>/Ext/Module.bsl`.
- ❌ **Вызывать экспортные методы из `#Область СлужебныйПрограммныйИнтерфейс` или `#Область СлужебныеПроцедурыИФункции` как стабильный API.** Эти области помечены как внутренние; обратная совместимость между минорными версиями БСП не гарантируется. Использовать допустимо только если в `#Область ПрограммныйИнтерфейс` нет аналога, и с явной отметкой в коде.
- ❌ **Реализовывать в `#Область ПрограммныйИнтерфейс` чужого модуля своё поведение.** Это «область программного интерфейса», метод там предназначен для вызова, а не для переопределения. Переопределение делается только через одноимённый модуль с суффиксом `Переопределяемый` (`ОбщегоНазначенияПереопределяемый`, `ПодключаемыеКомандыПереопределяемый`).
- ❌ **Дублировать функциональность БСП собственной реализацией.** Если БСП уже предоставляет метод (например, `ОбщегоНазначения.СообщитьПользователю`, `ПодключаемыеКоманды.ПриСозданииНаСервере`, `УправлениеПечатью.ВыполнитьКомандуПечати`) — не писать свою обёртку. Это ломает интеграцию с другими подсистемами БСП (журнал регистрации, замеры производительности, контроль прав).
- ❌ **Использовать `//`-комментарии как region markers вместо `#Область`.** `#Область` — это директива препроцессора, валидная конструкция языка 1С, видимая Конфигуратору как структурный блок. `//` — это обычный однострочный комментарий, который игнорируется при компиляции и не образует области.
- ❌ **Считать `БезопасноеХранилище` общим модулем.** Это регистр сведений (`БезопасноеХранилищеДанных`). Работа с ним — через методы серверного модуля `ОбщегоНазначения`, а не через прямой `БезопасноеХранилище.Записать(...)`.
- ❌ **Загружать огромный файл документации БСП целиком, когда достаточно посмотреть исходники в CommonModules/.** Для поиска API подсистемы открывать `NNN_интерфейс.md` соответствующей подсистемы (если документация доступна) либо смотреть исходники в `CommonModules/`.

Для поиска методов/модулей в выгрузке конфигурации используйте python scripts/bsp_core_search.py method <имя>.
