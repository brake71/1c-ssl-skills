
# BSP Classifiers (Производственные календари и графики работы)

Скил по подсистемам «КалендарныеГрафики», «ГрафикиРаботы» и связанной сущности «ПроизводственныйКалендарь». Покрывает расчёт бизнес-дат, подсчёт рабочих дней, определение ближайших рабочих дат, заполнение реквизита производственного календаря в форме и расписание работы на период.

## When to use

- Нужно рассчитать дату, отстоящую от заданной на N рабочих дней по производственному календарю или графику работы (плановый срок поставки, оплаты, отгрузки).
- Нужно посчитать количество рабочих дней между двумя датами по конкретному графику (срок задолженности, длительность этапа).
- Нужно сдвинуть дату к ближайшему рабочему дню с учётом нерабочих периодов (например, перенести дату сдачи отчётности с выходного на пятницу или с праздничного периода на ближайший рабочий день).
- Нужно определить основной производственный календарь, используемый в учёте, или заполнить реквизит «ПроизводственныйКалендарь» в форме документа/справочника.
- Нужно составить расписание работы (время начала и окончания) на даты, включённые в график работы, для расчёта трудозатрат.
- Нужно обновить данные зависимых от производственного календаря объектов (регистров сведений, графиков работы подразделений) после изменения производственного календаря.

## Не использовать, если

- Нужна загрузка производственного календаря из внешнего сервиса 1С (классификатора) — это делает подсистема «Работа с классификаторами» / `ИнтернетПоддержкаПользователей`, а не «КалендарныеГрафики». Загрузка вызывается автоматически подпиской на классификаторы; прикладному коду достаточно дождаться завершения фонового обновления.
- Нужно посчитать разность двух произвольных дат в календарных днях — это `РазностьДат(Дата1, Дата2, День)` платформы 1С или `ОбщегоНазначения` утилиты, не производственный календарь.
- Нужно табелировать учёт рабочего времени (табель Т-13), рассчитывать нормы часов по графику или формировать документы «Табель» — это прикладная подсистема учёта зарплаты, не «КалендарныеГрафики» / «ГрафикиРаботы».
- Нужно просто записать или прочитать ссылку на элемент справочника `ПроизводственныеКалендари` / `Календари` без расчёта дат — стандартные методы `Справочники.<Имя>.НайтиПоНаименованию(...)` и т. п.
- Нужен справочник банков, валют, ОКВЭД, ОКПО — это другие классификаторы (см. roadmap backlog, отдельный L3-скил `bsp-currencies-banks`).

## Core concepts

### Три сущности: календарь, график, расписание

В подсистеме три логически разных понятия, которые легко спутать:

- **Производственный календарь** (`СправочникСсылка.ПроизводственныеКалендари`) — нормативный документ: какие дни в году рабочие, предпраздничные, выходные, праздничные. Ведётся централизованно (часто автоматически обновляется из сервиса 1С). Один календарь в стране по умолчанию, может быть несколько (по регионам).
- **График работы** (`СправочникСсылка.Календари`) — производный объект: настраивается на основании производственного календаря (какие дни недели рабочие, цикличность), но может иметь индивидуальные исключения (ручные изменения). Используется для расчёта дат по конкретному подразделению / должности / виду операции.
- **Расписание работы** — таблица вида «дата — время начала — время окончания» на каждый день, включённый в график. Получается из графика с учётом производственного календаря.

Производственный календарь — это **справочник**, не подсистема верхнего уровня. Подсистема-владелец — `КалендарныеГрафики`. Графики работы выделены в отдельную подсистему `ГрафикиРаботы`, так как у них свой набор объектов (`Справочник.Календари`, `РегистрСведений.КалендарныеГрафики`, `РегистрСведений.РучныеИзмененияГрафиковРаботы`, обработка `ЗаполнениеГрафиковРаботы`).

### Где живут модули

- `CommonModules/КалендарныеГрафики/Ext/Module.bsl` — **основной** модуль программного интерфейса. Методы расчёта дат, ближайших рабочих дней, заполнения форм. Принимает как `СправочникСсылка.ПроизводственныеКалендари`, так и `СправочникСсылка.Календари` (внутри сам определяет, в какую подсистему делегировать, если передан график).
- `CommonModules/КалендарныеГрафикиСлужебный/Ext/Module.bsl` — служебный слой: обновление данных, зависимых от календарей, обработчики регламентных заданий. ⚠️ обратная совместимость не гарантируется; вызывается преимущественно кодом БСП, не прикладным.
- `CommonModules/КалендарныеГрафикиПереопределяемый/Ext/Module.bsl` — крючки для прикладного кода: переопределение реакции на обновление календаря.
- `CommonModules/ГрафикиРаботы/Ext/Module.bsl` — программный интерфейс по графикам работы (`СправочникСсылка.Календари`): `ДатаПоГрафику`, `БлижайшиеДатыВключенныеВГрафик`, `РасписанияРаботыНаПериод`. Календарные графики сами делегируют сюда при работе с графиками.
- `CommonModules/ГрафикиРаботыКлиент/Ext/Module.bsl` — клиентские утилиты для работы с коллекциями строк графика (`ВосстановитьПорядокСтрокКоллекцииПослеРедактирования`, `ВставитьВФиксированноеСоответствие`, `УдалитьИзФиксированногоСоответствия`). Для расчёта дат не используется.

Регистры сведений, лежащие в основе расчётов:

- `InformationRegisters/ДанныеПроизводственногоКалендаря` — собственно данные календаря по дням (рабочий / предпраздничный / праздник / выходной, дата переноса).
- `InformationRegisters/КалендарныеГрафики` — собственно дни, включённые в конкретный график работы (`СправочникСсылка.Календари`).
- `InformationRegisters/РучныеИзмененияГрафиковРаботы` — индивидуальные отклонения от шаблона.
- `InformationRegisters/ПериодыНерабочихДнейКалендаря` — особые нерабочие периоды (например, нерабочие дни, установленные указами президента). Учитываются в `БлижайшиеРабочиеДаты` при флаге `УчитыватьНерабочиеПериоды = Истина`.

### Доступность методов

Методы `КалендарныеГрафики` и `ГрафикиРаботы` доступны **на Сервере, в Толстом клиенте и во Внешнем соединении**. С тонкого клиента напрямую не вызываются — нужно либо вызывать серверный код через форму (`&НаСервере`), либо использовать `*ВызовСервера` обёртку (для подсистемы «КалендарныеГрафики» отдельного модуля `КалендарныеГрафикиВызовСервера` нет — делегируйте в свой серверный общий модуль). Модуля `КалендарныеГрафикиКлиент` в 3.1.11 нет.

### Виды дней

`Перечисление.ВидыДнейПроизводственногоКалендаря` имеет значения: `Рабочий`, `Предпраздничный`, `Праздник`, `Суббота`, `Воскресенье`, `НеУказан`. Методы `БлижайшиеРабочиеДаты` и `ДатаПоКалендарю` считают рабочими днями `Рабочий` и `Предпраздничный` (предпраздничный — укороченный рабочий день, но он входит в подсчёт).

### Функциональная опция «Несколько производственных календарей»

`ПолучитьФункциональнуюОпцию("ИспользоватьНесколькоПроизводственныхКалендарей")`:

- `Ложь` — в системе ровно один производственный календарь, метод `ОсновнойПроизводственныйКалендарь` возвращает его.
- `Истина` — календарь подбирается по региону (КПП организации), у одного и того же юрлица могут быть разные календари для разных обособленных подразделений.

Метод `ЗаполнитьПроизводственныйКалендарьВФорме` инкапсулирует эту логику: при `Ложь` ставит единственный календарь, при `Истина` — региональный по переданному КПП.

## Patterns

### 1. Расчёт плановой даты с учётом выходных и праздников

```bsl
// Плановая дата отгрузки: текущая дата + 5 рабочих дней по производственному календарю РФ
Календарь = КалендарныеГрафики.ОсновнойПроизводственныйКалендарь();
Если Календарь = Неопределено Тогда
    ПлановаяДата = ТекущаяДатаСеанса() + 5 * 86400;  // fallback: календарные дни
Иначе
    ПлановаяДата = КалендарныеГрафики.ДатаПоКалендарю(Календарь, ТекущаяДатаСеанса(), 5);
КонецЕсли;
```

`ДатаПоКалендарю` корректно пропускает субботы, воскресенья, праздники и предпраздничные дни (последние — укороченные, но считаются рабочими для подсчёта).

### 2. Сдвиг нескольких связанных дат (каждая от предыдущей)

```bsl
// Цепочка дат: 0 -> +3 -> +5 -> +7 рабочих дней от предыдущей (стадии согласования)
МассивДней = Новый Массив;
МассивДней.Добавить(3);
МассивДней.Добавить(5);
МассивДней.Добавить(7);

МассивДат = КалендарныеГрафики.ДатыПоКалендарю(
    Календарь,
    ДатаСтарта,
    МассивДней,
    Истина);  // <- РассчитыватьСледующуюДатуОтПредыдущей
// МассивДат[0] = ДатаСтарта + 3 рабочих дня
// МассивДат[1] = МассивДат[0] + 5 рабочих дней
// МассивДат[2] = МассивДат[1] + 7 рабочих дней
```

Параметр `РассчитыватьСледующуюДатуОтПредыдущей = Истина` превращает массив в «лесенку»: каждая следующая дата считается от предыдущей, а не от `ДатаОт`. Полезно для стадийных процессов (согласование → проверка → подписание).

### 3. Подсчёт рабочих дней в периоде (срок задолженности)

```bsl
// Сколько рабочих дней между датой отгрузки и текущей датой
ДнейПросрочки = КалендарныеГрафики.РазностьДатПоКалендарю(
    Календарь,
    ДатаОтгрузки,
    ТекущаяДатаСеанса());
```

Метод всегда возвращает положительное число, даже если `ДатаОкончания < ДатаНачала` (знак нормализуется внутри).

### 4. Ближайший рабочий день с учётом нерабочих периодов

```bsl
// Дата сдачи отчётности выпала на праздничный период — сдвигаем на ближайший рабочий
Параметры = КалендарныеГрафики.ПараметрыПолученияБлижайшихРабочихДат(Календарь);
Параметры.ПолучатьПредшествующие = Ложь;        // вперёд, не назад
Параметры.УчитыватьНерабочиеПериоды = Истина;   // учесть особые нерабочие дни (указы)

МассивИсходныхДат = Новый Массив;
МассивИсходныхДат.Добавить(ДатаСдачи);

Соотв = КалендарныеГрафики.БлижайшиеРабочиеДаты(Календарь, МассивИсходныхДат, Параметры);
НоваяДатаСдачи = Соотв[ДатаСдачи];  // ключ — исходная дата
```

Если исходная дата уже рабочая и не попадает в нерабочий период — метод вернёт её же.

### 5. Расписание работы на день (для трудозатрат)

```bsl
// Получить время начала и окончания работы по графику склада на 2024-03-15
Графики = Новый Массив;
Графики.Добавить(ГрафикСклада);

ТаблицаРасписаний = ГрафикиРаботы.РасписанияРаботыНаПериод(
    Графики,
    ДатаНачала,
    ДатаОкончания);
// Колонки: ГрафикРаботы, ДатаГрафика, ВремяНачала, ВремяОкончания
```

⚠️ Метод требует, чтобы подсистема `ГрафикиРаботы` была подключена в конфигурации. `КалендарныеГрафики.РасписанияРаботыНаПериод` — обёртка, которая делегирует в `ГрафикиРаботы` и бросает исключение, если подсистема отсутствует.

## Anti-patterns

### ❌ Считать рабочие дни через цикл по календарю

```bsl
// ❌ Платформенный цикл считает КАЛЕНДАРНЫЕ дни, не рабочие
ПлановаяДата = ТекущаяДатаСеанса();
Добавлено = 0;
Пока Добавлено < 5 Цикл
    ПлановаяДата = ПлановаяДата + 86400;
    Если ДеньНедели(ПлановаяДата) < 6 Тогда  // пн-пт
        Добавлено = Добавлено + 1;
    КонецЕсли;
КонецЦикла;
```

```bsl
// ✅ Производственный календарь учитывает праздники и предпраздничные дни
ПлановаяДата = КалендарныеГрафики.ДатаПоКалендарю(Календарь, ТекущаяДатаСеанса(), 5);
```

Цикл не учитывает праздничные дни, переносы выходных, нерабочие периоды по указам. Игнорирует «плавающие» выходные (например, 8 марта иногда выпадает на воскресенье — официальный выходной сдвигается на понедельник).

### ❌ Хранить ссылку на производственный календарь в константе / реквизите и считать, что он «единственный»

```bsl
// ❌ В системе с несколькими календарями константа вернёт «не тот»
Календарь = Константы.ОсновнойПроизводственныйКалендарь.Получить();
```

```bsl
// ✅ Метод учитывает функциональную опцию «Несколько производственных календарей»
Календарь = КалендарныеГрафики.ОсновнойПроизводственныйКалендарь();
```

При включённой опции «Несколько календарей» константы может не быть — метод `ОсновнойПроизводственныйКалендарь` возвращает «первый попавшийся», а в реальности подразделения работают по разным календарям. Корректный путь — `ЗаполнитьПроизводственныйКалендарьВФорме` с КПП подразделения.

### ❌ Смешивать `СправочникСсылка.ПроизводственныеКалендари` и `СправочникСсылка.Календари`

```bsl
// ❌ БлижайшиеРабочиеДаты принимает только производственный календарь
Соотв = КалендарныеГрафики.БлижайшиеРабочиеДаты(ГрафикРаботы, МассивДат);
//                       ^^^^^^^^^^^^^^ — может прийти СправочникСсылка.Календари
//                       Метод бросит исключение «ПроверитьПараметр»
```

```bsl
// ✅ Для графика работы — свой метод
Соотв = ГрафикиРаботы.БлижайшиеДатыВключенныеВГрафик(ГрафикРаботы, МассивДат);
```

`КалендарныеГрафики.ДатаПоКалендарю` принимает оба типа (внутри сам переключается на `ГрафикиРаботы.ДатаПоГрафику` при необходимости), но `БлижайшиеРабочиеДаты` — **только** `ПроизводственныеКалендари`. Это частая ошибка миграции кода.

### ❌ Вызывать `КалендарныеГрафикиСлужебный.*` для разовых пересчётов

```bsl
// ❌ ⚠️ Служебный API: предназначен для внутреннего обновления данных БСП
// Может измениться в любой минорной версии БСП
КалендарныеГрафикиСлужебный.ОбновитьДанныеЗависимыеОтПроизводственныхКалендарей(Условия, Идентификатор);
```

```bsl
// ✅ Если нужна логика «реакция на обновление календаря» — переопределите крючок
// в КалендарныеГрафикиПереопределяемый.ПриОбновленииПроизводственныхКалендарей
// Для разовых расчётов — используйте публичные методы КалендарныеГрафики.*
```

Служебные модули обновляют данные **всех** зависимых подсистем разом. Прикладной код обычно не должен ими пользоваться напрямую — для этого есть переопределяемые модули и регламентные задания.

### ❌ Запрашивать календарь в форме через `РеквизитФормыВЗначение` и не учитывать КПП

```bsl
// ❌ Заполнение «по умолчанию» без учёта региона
ЭтаФорма.Объект.ПроизводственныйКалендарь = КалендарныеГрафики.ОсновнойПроизводственныйКалендарь();
```

```bsl
// ✅ Метод сам подберёт региональный календарь по КПП, если в системе их несколько
КалендарныеГрафики.ЗаполнитьПроизводственныйКалендарьВФорме(
    ЭтаФорма,
    "Объект.ПроизводственныйКалендарь",
    КПП);
```

`ЗаполнитьПроизводственныйКалендарьВФорме` инкапсулирует обе ветки функциональной опции. Если КПП не передан при `ИспользоватьНесколькоПроизводственныхКалендарей = Истина` — будет взят «основной», что для обособленного подразделения скорее всего неверно.

### ❌ Использовать устаревший `КалендарныеГрафики.ДатыБлижайшихРабочихДней`

```bsl
// ❌ Метод помечен как устаревший (#Область УстаревшиеПроцедурыИФункции)
// Сейчас следует использовать: БлижайшиеРабочиеДаты или БлижайшиеДатыВключенныеВГрафик
Соотв = КалендарныеГрафики.ДатыБлижайшихРабочихДней(График, МассивДат, Ложь, Истина);
```

```bsl
// ✅ Стабильный API
Соотв = КалендарныеГрафики.БлижайшиеРабочиеДаты(Календарь, МассивДат);
```

Устаревший метод оставлен для обратной совместимости, но сигнатура длиннее (5 параметров вместо 3), и он делегирует в новые методы внутри. Не усложняйте код.

Для поиска методов/модулей в выгрузке конфигурации используйте python scripts/bsp_data_search.py method <имя>.
