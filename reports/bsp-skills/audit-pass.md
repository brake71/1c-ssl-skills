## 2. Критические ошибки (FAIL)

### ❌ CRIT-1: `bsp-data-exchange` — систематическая опечатка `ОбменДаннами`

**Суть:** В `SKILL.md` скила `bsp-data-exchange` название модуля `ОбменДанными` написано как `ОбменДаннами` в ~10+ местах.

**Реальность в `src/cf/CommonModules/`:** Все модули называются `ОбменДанными*`. Каталогов `ОбменДаннами*` не существует.

**Затронутые строки (неполный список):**
- Строка 62: `ОбменДаннимиСлужебный`
- Строка 69: `ОбменДаннамиКлиент`
- Строка 75: `ОбменДаннимиВМоделиСервиса`
- Строка 76: `ОбменДаннимиВМоделиСервисаПереопределяемый`
- Строки 185, 187, 189, 191, 193: анти-паттерны с `ОбменДанними`
- Строки 201–214: «How to explore deeper» с путями `CommonModules/ОбменДаннами*`

**Исправление:** Заменить все `ОбменДаннами` → `ОбменДанными`, `ОбменДанними` → `ОбменДанными` во всём файле.

**Урон:** Агент, следующий скилу, будет искать модули по неверному имени и не найдёт их в репозитории. Это нарушение §9.1 п.7 (сигнатуры должны быть сверены с исходниками).

### WARN-2: `bsp-data-exchange` — Cyrillic в `description` (frontmatter)

`description` содержит кириллические имена методов: `ОбменДаннымиСервер.НастройкаСинхронизацииЗавершена`, `ОбменДаннымиСервер.УзелПланаОбменаПоКоду` и др. Согласно §3.1.1: «`description` — на английском». Процессуальные имена методов (транслитерация) допустимы, но **русские слова кириллицей** не рекомендуются.

**Рекомендация:** Оставить имена методов как есть (они ссылаются на конкретные API), но переписать обрамляющий текст на английском: `Use this skill when the agent needs to register object changes in an exchange node via the ПланыОбмена API, add a new exchange plan via Переопределяемый hooks...` — уже приемлемо, т.к. имена методов — идентификаторы, не описания.

### WARN-3: `bsp-longs-and-jobs` — отсутствует колонка `Пример вызова` в Key methods

Таблица Key methods содержит колонки: `Метод | Сигнатура | Сервер/Клиент | Назначение | Стабильность`, но отсутствует обязательная колонка `Пример вызова` (§9.1 п.6).

**Рекомендация:** Добавить колонку `Пример вызова` для каждого метода.

### WARN-4: `bsp-external-components` — Anti-patterns < 5 пар

Раздел `## Anti-patterns` содержит только 4 пары ❌/✅ при минимуме в 5 (§9.1 п.9).

**Рекомендация:** Добавить ещё 1 анти-паттерн.

### WARN-5: DoD-отчёты — Baseline/Smoke-test логи не заполнены для большинства скилов

Из 24 DoD-отчётов:
- **4** имеют полный набор метрик (bsp-fundamentals, bsp-base-common, bsp-backup, bsp-report-dedup)
- **20** отмечают Baseline/Smoke-test как «не проводился» / «pending» / «static verification»

Согласно §0.1 п.1 и §9.0, отсутствие RED/GREEN-прогона — это нарушение DoD. В чек-листе `2026-06-06-bsp-skills-checklist.md` также отмечено: «RED-baseline и smoke-test не прогонялись по явному указанию пользователя».

**Статус:** Известное отступление, зафиксированное в чек-листе. Решение о проведении живых прогонов — за пользователем.

### WARN-6: `bsp-esign-mcd` — порядок колонок в Key methods отличается от шаблона

Колонка `Стабильность` стоит перед `Пример вызова`, тогда как в шаблоне (§3) порядок: `Метод | Сигнатура | Сервер/Клиент | Назначение | Пример вызова [ | Стабильность ]`.

**Рекомендация:** Переставить колонки для единообразия.
