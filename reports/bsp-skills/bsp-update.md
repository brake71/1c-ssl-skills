**Ожидаемый RED-результат (без скила, по аналогии с §0.1 п.6 плана):**
- Агент, скорее всего, напишет обработчик через «свой» шаблон — `ДокументОбъект.Записать()` напрямую внутри обработчика (типичная ошибка: триггерится бизнес-логика и регистрация обмена).
- Агент может попытаться «изобрести» модуль `ОбновлениеИнформационнойБазыСервер` (не существует; служебная ошибка по аналогии с `РегламентныеЗадания` без суффикса).
- Агент может вызвать `ОбновлениеИнформационнойБазыПереопределяемый.ПередОбновлениемИнформационнойБазы()` — **ошибка**, модуль содержит только замещаемые процедуры.
- Агент может не использовать `ОбновлениеИнформационнойБазы.ЗаписатьДанные` / `ЗаписатьОбъект` — обёртки для записи из обработчика.
- Агент может не знать про `ОбновлениеКонфигурации.ИсправленияИзменены` — обязательный шаг перед `ВыполнитьОбновлениеИнформационнойБазы`.

**Ожидаемое решение:** Case 1 из §9.0 — LLM без скила **не знает** идиоматический шаблон регистрации обработчика, не различает стабильный и служебный API, и использует прямые методы записи. Скил оправдан.

**Задачи 2 и 3 (для следующей итерации RED-прогона):**

**Дополнительные gotchas, зафиксированные в `## Core concepts`:**

- ⚠️ `ОбновлениеИнформационнойБазыСервер` (с суффиксом) **не существует** — серверный модуль называется `ОбновлениеИнформационнойБазы` без суффикса. По аналогии с `РегламентныеЗаданияСервер` (тоже с суффиксом), но `ОбновлениеИнформационнойБазы` — исключение.
- ⚠️ `ОбновлениеКонфигурацииПереопределяемый` и `ОбновлениеКонфигурацииСлужебный` **не существуют** — в отличие от семейства `ОбновлениеИнформационнойБазы*`, у `ОбновлениеКонфигурации` нет ни переопределяемых, ни служебных модулей.
- ✅ `ОбновлениеКонфигурации` — Сервер + Внешнее соединение + Толстый клиент (без `СерверCall`), весь стабильный API в нём.
