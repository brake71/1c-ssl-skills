# AGENTS.md — Инструкции для AI агентов

## Назначение репозитория

Документация **1С:БСП** (Библиотека стандартных подсистем) в Markdown, выгруженная с
официального сайта. **Основная рабочая версия — 3.1.11** (`bsp3111_md/`).
`bsp321_md/` — справочная копия версии 3.2.1, структура и набор файлов отличаются
(3046 vs 2911 файлов), взаимные ссылки между версиями **не поддерживаются**.

В `.claude/skills/` ведётся разработка скилов для AI агента, работающего с БСП 3.1.11.

## Структура

```
bsp3111_md/          # ~3000 файлов, основная версия
  index.md           # оглавление на ~4700 строк — единая точка входа
  001_…md            # Глава 1. Состав библиотеки
  002_…md            # Глава 2. Инструкция по внедрению
  003_…md            # Глава 3. Настройка и использование подсистем (1.9M, не читать целиком)
  004_…md            # Глава 4. Программный интерфейс (оглавление-стаб, см. ниже)
  005_*.md …         # документация подсистем и API
bsp321_md/           # справочная копия 3.2.1 (использовать только для сравнения)
.claude/skills/      # скилы, которые пишет/правит агент (SKILL.md + опционально references/, README.md, LICENSE)
src/                 # выгрузка исходного кода БСП (см. ниже), только чтение
```

## Исходный код БСП (`src/`)

`src/` — каталог с выгрузкой конфигурации **самой БСП** в формате XML
(стандартная выгрузка 1С, в одном корне `cf/`). Это локальный артефакт
разработчика, в `.gitignore`, в коммиты попадать не должен (~271M).

Зачем нужен: `bsp3111_md/` — это **документация** БСП (как пользоваться).
`src/cf/` — это **исходный код** БСП (как устроено внутри). Агенту нужен
доступ к обоим слоям, потому что документация неполна/неточна, а исходный
код всегда актуален. Сверяясь с `src/`, можно:

- проверить, что функция/процедура **реально существует** в указанном общем
  модуле и имеет ожидаемую сигнатуру;
- найти, в каком именно модуле БСП определён нужный экспорт (`CommonModules/`,
  `InformationRegisters/`, `Catalogs/` и т.д.);
- увидеть фактические имена параметров и типы возвращаемых значений, когда
  документация обтекаема;
- проследить внутренние вызовы и зависимости между подсистемами.

Структура:

```
src/
  cf/                # выгрузка конфигурации БСП 3.1.11 (271M)
                     #   Catalogs/, Documents/, CommonModules/,
                     #   InformationRegisters/, Subsystems/, ... — стандартные
                     #   объекты метаданных в виде XML
                     #   в т.ч. ~35 подсистем и ~1600 общих модулей
```

Правила работы с `src/`:

- **Только чтение.** Редактирование, удаление, перемещение файлов внутри
  `src/` запрещено — это снимок внешней выгрузки.
- Не пытайтесь коммитить `src/`: каталог в `.gitignore`.
- Версия выгрузки должна соответствовать `bsp3111_md/` (3.1.11). Если при
  сверке окажется, что в выгрузке код другой версии — сообщите пользователю,
  не подгоняйте факты молча.
- При поиске факта: сначала документация БСП (`bsp3111_md/`), затем
  подтверждение по исходникам в `src/cf/`. Если нашли противоречие —
  сообщите пользователю, не выбирайте молча один из источников.

## Соглашения по именованию файлов

- `NNN_название_на_русском.md` — трёхзначный префикс для упорядочивания, далее
  строчными буквами, слова через `_`. Пример: `008_кодрегионапонаименованию.md`.
- Имена функций/процедур 1С в Markdown идут **в именительном падеже без пробелов**
  (`КодРегионаПоНаименованию`), русский синтаксис языка 1С (не английский).

## Структура документации по подсистеме

Типичный блок подсистемы состоит из трёх слоёв (на примере «Адресный классификатор»):

1. `005_адресный_классификатор.md` — заголовок подсистемы, иногда пустой.
2. `006_интерфейс.md` — **только список имён** функций/процедур подсистемы, без описаний.
3. `007_*`, `008_*`, … — отдельный файл на каждую функцию/процедуру. Содержит
   секции: `Синтаксис`, `Параметры`, `Возвращаемое значение`, `Пример вызова`,
   `Доступность` (Сервер/Толстый клиент/Внешнее соединение).

> `004_глава_4_программный_интерфейс.md` — это **оглавление-стаб**: перечисление
> подсистем без API. Реальные сигнатуры и описания API лежат в файлах
> `006_интерфейс.md` каждой подсистемы и далее. Не ищите API в Главе 4.

## Навигация и чтение

- **Стартовая точка** — `bsp3111_md/index.md`. Все ссылки в нём указывают на
  `NNN_*.md#issoglN_…`; якоря сгенерированы парсером, ломать их руками не нужно.
- При поиске конкретной функции сначала открывайте `NNN_интерфейс.md` нужной
  подсистемы (например, `006_интерфейс.md` для Адресного классификатора), затем
  переходите к файлу с её именем.
- `003_…md` огромен (1.9M). **Не загружайте его целиком** — пользуйтесь
  `Grep`/`offset`/`limit` или открывайте `index.md` и идите по якорю.
- `bsp321_md/` используйте только когда явно нужна версия 3.2.1 (свежие
  подсистемы/функции, отсутствующие в 3.1.11). Файлы и индексы не пересекаются
  с 3.1.11 — не пытайтесь делать зеркальные ссылки.

## Работа со скилами в `.claude/skills/`

- Формат скила: `<name>/SKILL.md` + опционально `references/`, `README.md`,
  `LICENSE`, `icon.jpeg`. Шаблон и правила — в скиле
  `prompt-crafting-guide` (уже загружен в этой рабочей директории).
- Содержимое скила должно опираться на фактическую документацию БСП 3.1.11
  из `bsp3111_md/`, а не на общие знания о платформе 1С. Цитируйте
  подсистемы/функции по их реальным именам из `bsp3111_md/*интерфейс.md`.
- При подготовке скила дополнительно сверяйтесь с исходным кодом БСП
  в `src/cf/` (см. раздел «Исходный код БСП»): проверяйте реальные сигнатуры
  экспортных функций в `CommonModules/`, состав регистров в
  `InformationRegisters/`, перечень объектов подсистемы в `Subsystems/` и т.д.
  Это повышает фактическую точность скила.
- Перед добавлением нового скила — прочитать существующие
  `.claude/skills/*/SKILL.md`, чтобы выдержать стиль (frontmatter, разделы
  When to Use, формат ссылок).

## Что не трогать

- Файлы в `bsp3111_md/` и `bsp321_md/` — это выгрузка официальной документации.
  **Правка содержимого запрещена**, кроме случая исправления явных опечаток
  выгрузки (по согласованию с пользователем).
- `vault/`, `.kiro/`, `src/` перечислены в `.gitignore` — это локальные артефакты
  (knowledge vault, Kiro IDE, выгрузка исходного кода БСП), в коммиты
  не должны попадать.
- В репозитории **нет** пайплайна сборки/тестов/линтеров — никаких `npm test`,
  `pre-commit` и т. п. запускать не нужно.
